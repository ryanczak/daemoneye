# Product Requirements Document (PRD) Details for DaemonEye

This document specifies the functional and non-functional requirements for DaemonEye, building upon the Product Definition.

## 1. Functional Requirements

### 1.1 Daemon Process & tmux Integration

- **FR-1.1.1**: The application MUST run as a background daemon process, independent of any specific terminal emulator.
- **FR-1.1.2**: The application MUST use `tmux` as its presentation layer and session interaction mechanism.
- **FR-1.1.3**: The daemon MUST be capable of spawning new tmux panes within the user's active tmux session to present the AI interface.
- **FR-1.1.4**: The application MUST provide a CLI tool or tmux keybinding to trigger the AI agent, which communicates with the background daemon.
- **FR-1.1.5**: The daemon MUST append a structured JSON event record to `~/.daemoneye/var/log/events.jsonl` for every tool call execution event (approved, denied, or timed out).
- **FR-1.1.6**: `~/.daemoneye/` MUST serve as the shared runtime root for both the daemon process and the AI agent. All persistent data, configuration, logs, and agent-accessible resources MUST be organised under a single FHS-inspired directory tree initialised by `Config::ensure_dirs()` on every startup: `etc/` (user-editable configuration), `scripts/` and `runbooks/` (user-managed automation and procedures), `memory/` (persistent AI memory), `bin/` (executable symlinks and wrappers), `lib/` (shared SDK modules), `var/run/` (socket, schedule store, pane preferences), `var/log/events.jsonl` (structured event log), `var/log/` (daemon log, background-window output archives in `panes/`, live pipe-pane captures in `pipe/`), `etc/prompts/` and `var/log/sessions/` (daemon-generated prompts and conversation history). The AI agent's `read_file` tool MUST be blocked from the entire `~/.daemoneye/` tree except `var/log/panes/`, which MUST remain readable so the agent can page through archived background-command output. All other daemoneye-managed data MUST be accessed via dedicated AI tools (`read_script`, `read_runbook`, `read_memory`, etc.).
- **FR-1.1.8**: The daemon MUST always operate with a tmux session — there is no degraded "no session" mode. Session resolution at startup follows a two-step priority:
  (a) **Launched inside tmux** (`$TMUX` is set): the daemon MUST adopt the current tmux session via `detect_session()` (`tmux display-message -p '#S'`). The daemon does not own this session and MUST NOT attempt to recreate it if it is destroyed.
  (b) **Launched outside tmux** (e.g. as a systemd service): the daemon MUST use the configured session name — CLI `--session NAME` override, then `[daemon] tmux_session` in `config.toml`, with a default of `"daemoneye"`. If the named session already exists the daemon MUST adopt it (`session_exists()` check). If it does not exist the daemon MUST create it with `tmux new-session -d -s <name>`; a creation failure MUST be reported as a fatal startup error (`anyhow::bail!`). The session name MUST be overridable at launch with `daemoneye daemon --session NAME`.
  When the daemon owns a managed session and that session is destroyed, the daemon MUST recreate it automatically in the `NotifySessionClosed` handler (same `tmux new-session -d -s` call), repopulate `bg_session` and the session cache, and rely on the `after-new-session` global hook to reinstall per-session monitoring hooks in the new session.
  `daemoneye chat` invoked outside of tmux MUST detect the managed session name (CLI override, then `config.daemon.tmux_session`), open a new window named `chat` inside the managed session running the `daemoneye chat` binary, and exec-replace itself with `tmux attach-session -t <name>` so the user is dropped directly into the correct session; if the managed session does not exist, `daemoneye chat` MUST print a diagnostic message and exit with an error.
  This mode is the required configuration when running the daemon as a systemd user service (`Type=simple`) — the generated service file MUST use `daemoneye daemon --console` (no fork) so systemd tracks the correct PID. Since `"daemoneye"` is the default session name, no additional `[daemon]` configuration is needed for a standard headless deployment.

- **FR-1.1.7**: The `daemoneye setup` subcommand MUST support three optional overwrite flags for post-upgrade workflows. `--overwrite-bin` MUST copy the currently-running binary to `~/.daemoneye/bin/daemoneye`, replacing any previously installed version. `--overwrite-memory` MUST overwrite the six built-in knowledge memory files in `~/.daemoneye/memory/knowledge/` with the versions embedded in the current binary; user-created memory files in the same directory MUST NOT be removed or modified. `--overwrite-all` MUST behave as if both `--overwrite-bin` and `--overwrite-memory` were supplied, and MUST additionally overwrite `~/.daemoneye/etc/prompts/sre.toml` with the embedded built-in SRE prompt. The user's `config.toml` MUST never be overwritten by any of these flags. Without any flags, seeded files (binary, memories, prompt) are written only if they do not already exist (write-once semantics).

### 1.2 AI Agent Integration

- **FR-1.2.1**: The AI agent MUST act as an expert sysadmin and security expert, capable of determining the right tools to use.
- **FR-1.2.2**: When activated, the application MUST capture terminal context (visible output, backscroll history, environment state) from the currently active tmux pane and provide it to the AI agent on the **first turn** of each session. On subsequent turns the terminal snapshot is NOT automatically injected; the AI MUST request it explicitly via the `get_terminal_context` tool when it needs a fresh view of the screen. This eliminates repeated terminal snapshot accumulation in conversation history and reduces token consumption in conversational sessions. The captured context MUST include, for each pane: (a) the shell's current working directory (`#{pane_current_path}`); (b) the OSC terminal title (`#{pane_title}`) when set by a running application (e.g., vim, ssh, k9s) and distinct from the command name; (c) a `[dead: N]` annotation for background panes whose foreground process has exited with code N (remain-on-exit mode); when the pane has been idle since completion, the idle duration MUST be folded into the same tag as `[dead: N, idle Xs]` / `[dead: N, idle Nm]` / `[dead: N, idle NhNm]` so the AI can see both exit code and inactivity age in a single annotation. Non-active panes MUST be classified into three labeled categories: `[VISIBLE PANE]` (same tmux window as the chat interface), `[BACKGROUND PANE]` (daemon-launched `de-bg-*` / `de-sched-*` windows), and `[SESSION PANE]` (other user-created windows). This classification MUST use the cached `window_name` field with no additional tmux subprocess calls. Window-relative pane indices (`idx:N`) MUST NOT appear in individual pane headers; they are available exclusively in the `[PANE MAP]` line (see below). (d) a temporal activity annotation on non-active panes: `[active Xs ago]` when last activity was under 30 s ago, `[idle Nm]` when under 1 h, and `[idle NhNm]` otherwise — derived from `#{pane_activity}` (Unix timestamp of last keypress or output in the pane). (e) ANSI SGR foreground colour codes in the active pane content MUST be converted to semantic markers before the context is sent to the AI: red (codes 31, 91) → `[ERROR: text]`, yellow (33, 93) → `[WARN: text]`, green (32, 92) → `[OK: text]`; all other escape sequences MUST be stripped. This applies to both the pipe-log path (via `annotate_ansi()`) and the `capture-pane` fallback paths (via `capture_pane_with_escapes()` with `-e` flag, then `annotate_ansi()`). (f) when two or more tmux sessions exist, an `[OTHER SESSIONS]` block MUST be appended to the context snapshot listing each non-current session's name, window count, time since last activity, and attachment state — derived from `tmux list-sessions -F` (N16). This block MUST appear only when the snapshot already contains pane or environment data (i.e. MUST NOT convert a `(no terminal context available)` result into a non-empty snapshot). Additionally, the application MUST fetch high-signal tmux session environment variables via `tmux show-environment` filtered against an allowlist (AWS_PROFILE, AWS_DEFAULT_REGION, AWS_REGION, KUBECONFIG, KUBE_CONTEXT, KUBECTL_CONTEXT, VAULT_ADDR, DOCKER_HOST, DOCKER_CONTEXT, ENVIRONMENT, APP_ENV, NODE_ENV, RAILS_ENV, RACK_ENV, VIRTUAL_ENV, CONDA_DEFAULT_ENV, GOPATH, GOENV, JAVA_HOME, LANG, LC_ALL) and prepend them as a `[SESSION ENVIRONMENT]` block to every AI context snapshot. (g) a `[FOREGROUND TARGET] %N (cmd, 'window')` line MUST be prepended to the context block on the first turn and injected as a standalone line on every subsequent turn alongside `[PANE MAP]`. This line names the session's designated foreground pane — the exact `%N` the AI MUST pass to `run_terminal_command(background=false)` without deriving the target from topology. If the foreground target changes between turns (pane closed, user moved focus, `/pane` used), the daemon MUST send a `[Pane target changed]` `Response::SystemMsg` after `SessionInfo` so the model updates before generating its first token.
- **FR-1.2.3**: The AI MUST be able to analyze stack traces, crash logs, failing services, and security scan outputs to provide root cause analysis and remediation strategies.
- **FR-1.2.4**: The AI interacts directly with the user's active terminal using tmux session features. This allows the AI agent to "pair program" with the user. By hooking into the user's terminal session via tmux, the AI agent can execute commands, read output, and respond to system prompts with the user's permission.
- **FR-1.2.5**: The application MUST actively audit the system state (OS release, uptime, memory, load average, top CPU processes, shell environment, and shell history) once per session, cache it, and prepend this summary to the AI agent's context alongside the visible terminal buffer.
- **FR-1.2.6**: The daemon MUST detect its own hostname and whether the user's active tmux pane is connected to a remote host (via SSH or mosh). This execution context MUST be injected into the first-turn AI prompt so the AI understands which machine each execution mode (`background` vs `foreground`) will target.
- **FR-1.2.7**: The application MUST support two distinct command execution modes and the AI MUST be instructed to choose between them appropriately:
  - *Background mode*: Command runs in a dedicated tmux window (`de-bg-<session_name>-<YYYYMMDDHHMMSS>-<id_short>`) on the daemon host using the user's configured shell. The window is configured with `remain-on-exit on`. The command is wrapped with a shell-appropriate IPC notification (`cmd; __de_ec=$?; daemoneye notify complete <pane_id> $__de_ec <session>` for POSIX shells; `set __de_ec $status` for fish/csh) so the shell remains alive after the command finishes, keeping the window available for follow-up commands. The tool call returns **immediately** with the assigned pane ID; completion is detected asynchronously via two paths: (primary) the command wrapper sends the exit code via `daemoneye notify complete` IPC, broadcasting on `COMPLETE_TX` — window remains open for reuse; (fallback) the shell exits unexpectedly, `pane-died` hook fires and broadcasts on `BG_DONE_TX` — window is GC'd automatically. Upon completion the pane's entire scrollback history is logged to `~/.daemoneye/var/log/panes/` and a `[Background Task Completed]` context message (containing exit code and captured output) is injected into the AI session history. Up to 5 background windows persist per chat session; the oldest completed window is evicted when the cap is reached; all windows are closed when the session expires. The AI MUST be able to explicitly close a background window via a `close_background_window(pane_id)` tool call (no user approval required). The daemon MUST automatically garbage-collect background windows whose `completed_at` timestamp is more than 15 minutes old, checked on the 60 s session-cleanup cycle, to prevent window accumulation even if the AI never calls `close_background_window`. The application MUST share background commands with the user and gain approval before executing them.
  - *Foreground mode*: Command is injected into the user's active tmux pane via `send-keys`. Completion MUST be detected via a three-way branch keyed on the command type: (1) *Interactive commands* (`ssh`, `mosh`, `telnet`, `screen`, and similar session-hijacking tools) — the daemon MUST detect these via `is_interactive_command()` before injecting the command. It MUST wait up to 10 seconds for a shell-prompt pattern to appear in the pane output, then return immediately with an `[Interactive session started]` tool result identifying the destination host and instructing the AI to use `run_terminal_command(target_pane=...)` for subsequent commands. It MUST NOT wait for the interactive session to exit. Non-interactive sub-cases (`ssh host command`, `ssh -N`) are excluded and handled by path (2) or (3) as appropriate. (2) *Remote panes* (pane whose foreground process is `ssh` or `mosh-client`) — output-stability polling (two consecutive identical snapshots, 500 ms interval, 30 s timeout). (3) *Local panes* — event-driven `pane-title-changed` hook notification with `pane_current_command` polling fallback (45 s timeout). The user can see and interact with the command as it runs. If the user's pane is SSH'd to a remote host, the command executes on that remote host. If the AI does not explicitly set a target pane, the CLI client MUST perform automatic pane discovery before connecting: auto-select the sole sibling pane in the same window, present a numbered picker when multiple siblings exist, or offer to split the window side-by-side (default) or pick from other session panes when the chat pane is alone. The chosen pane MUST be persisted to `~/.daemoneye/var/run/pane_prefs.json` keyed by session name so the user is never prompted again after the first run. The persisted preference is always overridable by explicit AI instructions or by the `/pane %N` in-chat command. `/pane` (no argument) MUST list all targetable panes with the current target marked; `/pane %N` MUST validate the pane ID format (`%\d+`), update `SessionEntry.default_target_pane`, persist to `pane_prefs.json`, and confirm via `Response::PaneChanged`. *Pane ID validation*: when the AI passes a `target_pane` to `run_terminal_command(background=false)` that does not match the `%\d+` format (e.g. a bare index `"1"`), the daemon MUST reject the call immediately with a corrective error message naming the session's `default_target_pane` so the model can self-correct on the same turn without requiring a tool call cycle. *Exit code capture*: after detecting command completion, the daemon MUST read the tmux session environment key `DE_EXIT_<N>` (where N is the numeric pane ID, e.g. `DE_EXIT_3` for pane `%3`) via `tmux show-environment` to obtain the real exit code. This key is written by an optional shell hook (`PROMPT_COMMAND` / `precmd_functions`) that runs after every command completes. If the key is absent (hook not configured), the daemon MUST record exit code 0. The daemon MUST call `stats::finish_command(cmd_id, exit_code)` to update the recent-command record and increment the appropriate `COMMANDS_FG_SUCCEEDED` / `COMMANDS_FG_FAILED` counter. `daemoneye setup` MUST print the bash and zsh hook snippets so users can opt in.
- **FR-1.2.10**: The daemon MUST support scheduling commands to run once at a specified UTC datetime or repeatedly at a specified ISO 8601 interval (e.g. `PT5M`, `PT1H`, `P1D`). Scheduled jobs MUST persist across daemon restarts in `~/.daemoneye/var/run/schedules.json` using atomic writes. Each job MUST run in a dedicated tmux window pane (`de-sched-<YYYYMMDDHHMMSS>-<id_short>`), which MUST be destroyed on success and left in place on failure for inspection. Output is captured vis 'capture-pane' and the exit code is extracted natively via `pane_dead_status` and returned to the AI. Scheduled job panes are tracked via `pane-died` and `alert-bell` hooks. The AI MUST be able to create, list, cancel, and delete scheduled jobs via tool calls. The daemon MUST support a `ScheduleWritePrompt` approval gate that displays the full schedule content to the user before writing.
- **FR-1.2.11**: The daemon MUST support watchdog jobs that run a command on a schedule and pass the captured output to the AI for analysis using a named runbook (markdown file in `~/.daemoneye/runbooks/<name>.md`). Runbooks use YAML-style frontmatter to declare tags and knowledge memory references. The watchdog AI prompt MUST include any knowledge memory entries referenced in the runbook's `memories:` frontmatter field. When the AI detects an alert condition, it MUST send a `SystemMsg` notification to connected clients. If `[notifications] on_alert` is configured in `config.toml`, the daemon MUST invoke that shell command with `$DAEMONEYE_JOB` and `$DAEMONEYE_MSG` environment variables set.
- **FR-1.2.12**: The daemon MUST maintain a `~/.daemoneye/scripts/` directory of executable scripts (mode 0700). Scripts MAY be shell (`.sh`) or Python (`.py`); the daemon MUST execute them correctly regardless of extension based on their shebang line. The AI MUST prefer Python scripts for tasks involving data processing, JSON parsing, REST API calls, or logic with more than two stages of transformation, and MUST use shell scripts only for simple orchestration. The AI MUST be able to write new scripts and update existing ones, subject to a `ScriptWritePrompt` approval gate. The gate MUST:
  (a) display an ANSI-coloured diff before asking for confirmation — new scripts show all lines as green `+` additions; modifications show a unified diff with red `-` removed lines, green `+` added lines, and cyan `@@` hunk headers (3-line context, generated via `cli::diff::render_diff` backed by the `similar` crate);
  (b) offer three choices: `[y]es` (approve once), `[A]pprove for session` (auto-approve all future writes to this specific script name for the remainder of the chat session — diff is still shown, gate is bypassed), and `[N]o` (deny);
  (c) when session-approved, display `✓ auto-approved (session)` immediately after the diff without prompting.
  The AI MUST be able to list available scripts and read their contents without approval.
- **FR-1.2.13**: The daemon MUST provide a Knowledge System comprising three inter-related persistence layers accessible as AI tools:
  - *Runbook CRUD*: The AI MUST be able to `write_runbook` (approval-gated, validates required sections), `read_runbook`, `list_runbooks`, and `delete_runbook` (approval-gated, warns if active scheduled jobs reference it). Runbooks MUST use markdown format with YAML-style frontmatter (`tags`, `memories` fields). `delete_runbook` MUST check active jobs and include affected job names in the approval prompt. The `write_runbook` approval gate MUST follow the same diff-display and session-approval rules as `write_script` (FR-1.2.12): ANSI diff shown before prompting, `[y]es / [A]pprove for session / [N]o` options, per-runbook-name session approval tracked independently of script approvals.
  - *Persistent Memory*: The AI MUST be able to `add_memory`, `update_memory`, `delete_memory`, `read_memory`, and `list_memories` across three categories — `session` (auto-loaded at each turn, 32 KB cap, masking applied), `knowledge` (on-demand via runbook references or explicit reads), and `incident` (historical records, search-only). Memory operations MUST NOT require user approval. Memory entries MUST support structured YAML frontmatter fields: `tags` (retrieval terms and synonyms), `summary` (one-line description surfaced in `list_memories` output and contextual auto-search), `relates_to` (cross-references to other memory keys, runbooks, or scripts), `created`/`updated` (ISO timestamps), and `expires` (optional TTL for time-bounded facts). `update_memory` MUST perform partial updates — only the provided fields are changed, all other fields and body content are preserved — and MUST automatically set the `updated` timestamp. On the first AI turn the daemon MUST inject two knowledge blocks into the system prompt: (a) `## Available Knowledge` — a compact ≤1 KB manifest listing all stored runbooks, scripts, knowledge memories, and incidents; knowledge and incident entries MUST be shown in `key — summary [tags]` format when metadata is available; (b) `## Auto-loaded Knowledge` — up to 3 entries whose names, tags, or summary words match the first user query and visible pane content; the auto-search MUST also follow `relates_to` links on matched knowledge memories to include related entries up to the 3-item cap. Tool results that reference stored knowledge MUST be annotated with a `[Related knowledge: …]` hint line when the output contains terms matching stored entry names, tags, or summary words.
  - *Repository Search*: The AI MUST be able to `search_repository` across runbooks, scripts, all memory categories, and the last 10,000 lines of `events.jsonl`. Search MUST be case-insensitive, match both filenames and content, include configurable context lines around each match, and cap results at 50 entries.
  - *Terminal Context*: The AI MUST be able to call `get_terminal_context` to obtain a fresh snapshot of the current tmux session (active pane contents, background pane summaries, session topology, and environment). This tool requires no user approval. It MUST be the mechanism by which the AI accesses the terminal on turns after the first.
  - *Pane Discovery*: The AI MUST be able to call `list_panes` to obtain a compact table of all active panes in the session (pane ID, window name, foreground command, working directory, terminal title, synchronized and dead-process flags). The chat pane MUST be excluded from results. This tool MUST read from the in-memory SessionCache and MUST NOT spawn a tmux subprocess. Each entry MUST include the activity annotation (same `[active Xs ago]` / `[idle Nm]` / `[idle NhNm]` format used in the terminal context snapshot) so the AI can identify recently-used panes. The agent uses this to identify a pane by its characteristics before targeting it via `run_terminal_command(target_pane=...)`.
- **FR-1.2.9**: The chat input MUST operate in raw terminal mode to support readline-style editing. Users MUST be able to navigate the current input line with ←/→ arrow keys (and Home/End, Ctrl+A/E) and submit past entries with ↑/↓ arrow keys. The input MUST support in-line deletion (Backspace, Delete, Ctrl+K, Ctrl+U). The input area MUST word-wrap long text across multiple rows (up to a maximum of 5) rather than scrolling horizontally; the input box MUST grow upward dynamically as the buffer fills and collapse back to 1 row when text is deleted or on submission. The top border of the input frame MUST display the local `user@host` identity. History MUST persist for the lifetime of the chat session and reset when the session ends.

- **FR-1.2.14**: The daemon MUST optionally expose an HTTP webhook endpoint (configurable port, default 9393; bind address configurable via `[webhook] bind_addr`, defaulting to `"127.0.0.1"` so that the endpoint is not exposed to the network on internet-facing hosts; set to `"0.0.0.0"` to expose on all interfaces; disabled by default). It MUST accept Prometheus Alertmanager v4 format, Grafana unified alerting format, Grafana legacy webhook format, and a generic JSON fallback. Received alerts MUST be: (a) deduplicated within a configurable window by fingerprint (sourced from the payload or computed from sorted label key=value pairs); (b) masked for sensitive data via the existing `mask_sensitive()` filter before any further processing; (c) injected as `[Webhook Alert]` messages into all active AI session JSONL files; (d) displayed via `tmux display-message` in all active chat panes. When `[webhook] severity_threshold` is met, `fire_notification()` MUST be invoked. When `[webhook] auto_analyze = true` and a matching runbook exists (checked using kebab-case, lowercase, and exact-case variants of the alert name), a runbook-based AI analysis MUST run; if the AI response contains "ALERT" the analysis MUST be injected into sessions and chat panes. A configurable Bearer token MUST be validated when `[webhook] secret` is non-empty; requests without a valid token MUST be rejected with `401 Unauthorized`. The endpoint MUST return `200 OK` immediately (processing is asynchronous). A `GET /health` endpoint MUST return `"ok"` for liveness probes.

- **FR-1.2.15**: The daemon MUST detect client-attach and client-detach events via two global tmux hooks (`client-attached` and `client-detached`) installed at startup. On `client-detached`, the daemon MUST record `last_detach = now` and `messages_at_detach = messages.len()` on every active `SessionEntry` whose `tmux_session` field matches the detached session. On `client-attached`, the daemon MUST clear `last_detach` on matching entries. When the next `Ask` request arrives and `away_secs >= 30`, the daemon MUST scan `messages[messages_at_detach..]` for event messages (identified by the prefixes `[Background Task Completed`, `[Webhook Alert]`, `[Watchdog]`, `[Watch Pane`, `[Ghost Shell Started]`, `[Ghost Shell Completed]`, `[Ghost Shell Failed]`) and, if any are found, send a `Response::SystemMsg` containing a `[Catch-up] N events while you were away (Xm): …` summary immediately after `Response::SessionInfo`. The catch-up brief MUST be sent only once per detach cycle (i.e. `last_detach` MUST be cleared after generating the brief).

- **FR-1.2.16**: When the user's tmux environment contains two or more sessions, the daemon MUST include an `[OTHER SESSIONS]` block in every terminal context snapshot (both automatic first-turn and on-demand `get_terminal_context` calls). Each entry MUST include the session name, window count, time elapsed since the session's last activity (formatted as `Xm ago`, `XhYm ago`, etc.), and whether a client is currently attached. The block MUST be omitted when the snapshot has no pane or environment data (to prevent polluting the `(no terminal context available)` fallback path during tests or when the tmux session is unavailable).

- **FR-1.2.17**: The daemon MUST support Autonomous Ghost Shells for unattended alert response. A Ghost Shell: (a) is triggered by a webhook when `auto_analyze = true` and the matching runbook watchdog analysis emits `GHOST_TRIGGER: YES` (parsed by `parse_ghost_trigger()`, case-insensitive, last matching line wins; falls back to a legacy `ALERT` keyword check) and the runbook has `enabled: true` in its `ghost_config` frontmatter section (`ghost_config.enabled` in code); the watchdog analysis call MUST use `use_tools=false` so the LLM is constrained to plain-text output — Gemini MUST receive `toolConfig.functionCallingConfig.mode="NONE"`, Anthropic MUST omit the `tools` field, and OpenAI MUST set `tool_choice="none"` — preventing the model from accidentally emitting a function call instead of the `GHOST_TRIGGER` decision line; the outcome MUST be logged at INFO level with a `trigger_reason` string (`"GHOST_TRIGGER: YES"`, `"GHOST_TRIGGER: NO"`, `"legacy ALERT keyword"`, `"no GHOST_TRIGGER line and no ALERT keyword"`, `"empty_response — model returned no tokens"`, or `"api_error — response empty"`); (b) runs in a dedicated headless tmux window (`de-incident-<alertname>-<id>`) in the user's most-recently-attached session, validated via `session_exists()` before the AI loop begins; (c) is governed by a `GhostPolicy` with an OS-delegation trust model: non-sudo commands MUST always be allowed (the daemon runs as the same user, so OS file permissions are the boundary); `run_with_sudo` does NOT affect `GhostPolicy::is_safe()` — sudo commands are always denied unless the script basename is listed in the runbook's `auto_approve_scripts` frontmatter field AND a NOPASSWD sudoers rule has been installed via `daemoneye install-sudoers`; `run_with_sudo: true` only controls `GhostPolicy::resolve_command()` (see sub-item (f) below); when `run_with_sudo: false` (default), scripts in `auto_approve_scripts` run as the current user unless the ghost explicitly writes `sudo script.sh`; `GhostPolicy::is_safe()` MUST strip a leading `sudo` token before basename extraction so whitelisted scripts are recognised regardless of whether the AI emits them with or without a `sudo` prefix; (d) auto-denials of sudo commands (only applicable when `run_with_sudo: false`) MUST be logged at INFO level with the full command and whitelist contents so operators can diagnose policy mismatches; (e) enforces a two-level turn budget: the runbook frontmatter field `max_ghost_turns: N` (0 = daemon default) MUST be clamped to the daemon-wide hard ceiling configured as `max_ghost_turns` under the `[ghost]` section of `config.toml` (default 20); the ghost system prompt MUST include a `Turn Budget: N (hard limit)` line; a WARN is logged when the limit is reached; (f) supports a `run_with_sudo: true/false` runbook frontmatter field (default `false`); when `true`, `GhostPolicy::resolve_command()` MUST prepend `sudo` to the resolved script path so the ghost AI can write `script.sh` instead of `sudo script.sh`; `GhostPolicy::is_safe()` is NOT affected — the `auto_approve_scripts` whitelist check is always enforced regardless of this flag; the ghost system prompt MUST annotate pre-approved scripts with "(executed with sudo)" when the flag is set and MUST state `"Sudo commands require a pre-approved script via install-sudoers"` in all cases; (g) injects `[Ghost Shell Started]`, `[Ghost Shell Completed]`, `[Ghost Shell Failed]`, and `[Ghost Shell Skipped]` lifecycle events into all active sessions via `inject_ghost_event()` so they appear in catch-up briefs; `inject_ghost_event()` MUST also append a `ghost_lifecycle` record to `events.jsonl`; (h) tracks live ghost count via `GHOSTS_ACTIVE` atomic, reported in `daemoneye status` as `ghosts_active`; (i) MUST log structured records to `events.jsonl` for the full ghost lifecycle: `ghost_start` (session created — `session_id`, `alert_name`, `tmux_session`, `trigger`), `ghost_turn` (per tool-dispatching turn — `session_id`, `turn`, `tool_count`, `tools` array), `ghost_complete` (`session_id`, `turns_used`), and `ghost_error` (errors, timeouts, and max-turns exhaustion — `session_id`, `turn`, `error`); `daemon.log` MUST record per-turn INFO entries for each dispatched tool call including the command string; (j) MUST write the ghost session's initial user message to the session JSONL file (`~/.daemoneye/var/log/sessions/<session_id>.jsonl`) immediately upon session creation — before `trigger_ghost_turn` is called — so the file exists even if the ghost fails before completing any turn; (k) pane output from ghost background commands MUST be archived to `~/.daemoneye/var/log/panes/<win_name>.log` using the full pipe-pane log content when available (no scrollback truncation), falling back to `capture_pane_to_file` only when no pipe log exists.

- **FR-1.2.18**: The daemon MUST implement structured session digest compaction to preserve reasoning continuity across long sessions. When the in-memory message count reaches a configurable threshold (`DIGEST_THRESHOLD`, default 30), the daemon MUST replace the oldest messages with a structured `[Session Digest]` block — without requiring an additional LLM API call. The digest MUST be built by scanning `events.jsonl` filtered by session ID and session start time, tallying: commands executed (success/fail with names), files edited, cumulative token usage (prompt and completion), background window lifecycle (created/closed/active), webhook alerts received, and ghost shell activity (started/completed). The digest MUST also scan the filesystem for artifacts created or modified since session start: runbooks (by mtime in `~/.daemoneye/runbooks/`), scripts (by mtime in `~/.daemoneye/scripts/`), memories across all three categories (by mtime), and scheduled jobs (by `created_at` in `schedules.json`). `SessionEntry` MUST carry a `started_at: DateTime<Utc>` field set at session creation to anchor the time-range filter. After compaction the message layout MUST be `[first_message, digest_as_assistant, last N tail messages]` with the tail snapped to an even index to preserve user→assistant role alternation. When the threshold is not reached, the existing `trim_history()` tail-trim MUST apply unchanged. The digest function MUST log structured events to `events.jsonl` for its own workflow: `session_digest_start`, `session_digest_events_scanned`, `session_digest_artifacts_found`, and `session_digest_complete`.

- **FR-1.2.19**: The chat interface MUST provide an `/approvals` in-chat command for inspecting and managing session-level approval state:
  - `/approvals` (no argument) MUST print the current approval state for all five scopes: terminal commands (regular), terminal commands (sudo), per-script session approvals (names listed), per-runbook session approvals (names listed), and per-file-path session approvals (paths listed with `~`-relative display). Active approvals MUST be visually distinguished from inactive ones (e.g. ⚡ vs dimmed "off" / "none").
  - `/approvals revoke` MUST reset all session approvals to their default state (everything requires explicit confirmation) and update the status bar hint immediately. `/approvals revoke commands`, `/approvals revoke scripts`, `/approvals revoke runbooks`, and `/approvals revoke files` MUST each revoke only the named class and update the status bar.
  - Approval state MUST also reset on `/clear`, `/prompt`, `/refresh`, and Ctrl+C (interrupt during approval prompt).
  - The status bar hint MUST use count-based segments (e.g. `scripts: 2`, `files: 1`) rather than listing individual names, to prevent overflow on narrow terminals. When the full hint still overflows after dropping the model and token segments, it MUST be truncated with `…` before being dropped entirely.
  - The daemon MUST track write-approval and write-denial counts for scripts, runbooks, and file edits as persistent atomic counters since daemon start. These counts MUST be included in `Response::DaemonStatus` and displayed in the **Tooling** section of `daemoneye status` alongside the existing created/executed/deleted stats.

### 1.3 Prompt Library

- **FR-1.3.1**: The application MUST include a library of pre-defined prompts for common tasks.
- **FR-1.3.2**: Users MUST be able to create, save, and manage their own custom prompts. A `daemoneye prompts` subcommand MUST list available prompts with a brief preview of each prompt's system message. A `/prompt <name>` in-chat command MUST start a new session using the named prompt as the system message. *(Config and file loading are implemented; the subcommand and in-chat command are pending.)*
- **FR-1.3.3**: All user-defined and standard prompts MUST be stored under `~/.daemoneye/etc/prompts/`.

### 1.4 Authentication & Security

- **FR-1.4.1**: Sensitive data (passwords, secret keys, PII) MUST be masked or filtered from the terminal buffer before being transmitted to the AI API. The filter MUST cover at minimum: AWS access key IDs, PEM private key blocks, GCP service-account JSON private key fields, JWT bearer tokens, GitHub personal access tokens, database/broker connection URLs with embedded credentials, password/token/API-key assignments, URL query-param secrets, credit card numbers, and SSNs. Users MUST be able to extend the filter with additional patterns via `[masking] extra_patterns` in `config.toml`; built-in patterns MUST NOT be disableable. Redaction counts MUST be tracked per type using per-process atomic counters and exposed in `daemoneye status` under a **Redactions** section. All built-in counter types MUST always be displayed, including those with a zero count. Hits from user-configured `extra_patterns` MUST be aggregated into a separate `"User Defined"` counter distinct from all built-in type counters.
- **FR-1.4.2**: Users MUST have explicit controls over what terminal context is sent to the LLM.
- **FR-1.4.3**: When the AI proposes a background command that requires `sudo`, the daemon MUST detect the sudo password prompt in the tmux window via `capture-pane`. It MUST then send a `CredentialPrompt` IPC response to the client (echo disabled). The password MUST be injected into the background window via `send-keys` and MUST NOT be logged, stored on disk, or transmitted to the AI.
- **FR-1.4.4**: When the AI proposes a foreground command that requires `sudo`, the daemon MUST inject the command into the user's terminal pane and wait briefly for it to start. The daemon MUST then capture the pane output and check whether a sudo password prompt is present. If a password prompt is detected: the application MUST notify the user in the chat interface, make the target terminal pane active so the user can type their password, wait for the password to be entered and the command to complete, then switch focus back to the AI chat pane. If no password prompt is detected (e.g. a NOPASSWD sudoers rule or a still-valid credential timestamp), the application MUST proceed with a standard wait interval without switching panes.

### 1.5 Extensibility

- **FR-1.5.1**: The application MUST include a native plugin architecture for community extensions.
- **FR-1.5.2**: The application MUST allow plugins to hook into AI prompt lifecycles, and third-party APIs (e.g., AWS/GCP/Azure).

## 2. Non-Functional Requirements

### 2.1 Performance

- **NFR-2.1.1**: Capturing tmux buffers and transmitting to the AI MUST NOT block the user's terminal interaction.
- **NFR-2.1.2**: The daemon MUST be lightweight and consume minimal background system resources when idle.

### 2.2 Compatibility & Environment

- **NFR-2.2.1**: The application MUST run on standard modern Linux distributions (Ubuntu, Fedora, Arch, etc.).
- **NFR-2.2.2**: The application requires `tmux` to be available in the system PATH.

### 2.3 Usability

- **NFR-2.3.1**: The interaction with the AI agent inside the tmux pane MUST feel natural and responsive.
- **NFR-2.3.2**: The chat interface MUST provide a visual indicator (animated spinner) while waiting for the AI to respond.
- **NFR-2.3.5**: The chat interface MUST display a context-budget indicator in the bottom border of each user query box, showing `turn N · Xk / Yk tokens · Z% remaining`. The context window (Yk) MUST be determined from the configured model. The indicator MUST be color-coded by percentage consumed: dim when under 50 %, yellow at 50–74 %, bold red at ≥ 75 %. The value MUST persist across turns (carried from the previous AI response's `UsageUpdate`) so the user always sees an accurate budget before they submit the next query.
- **NFR-2.3.3**: Tool call prompts MUST clearly communicate which execution mode will be used and where the command will run (daemon host or user's terminal pane).
- **NFR-2.3.4**: The chat interface SHOULD use color and Unicode symbols to visually distinguish AI responses, tool calls, approvals, errors, and system messages.
