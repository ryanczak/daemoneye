## 2. Runtime Root (`~/.daemoneye/`)

`~/.daemoneye/` is the shared runtime root for both the **daemon process** and the **AI agent**. All configuration, managed data, logs, and agent-accessible files live under a single FHS-inspired tree:

| Path | Purpose |
|---|---|
| `etc/config.toml` | Daemon configuration (`[models.*]` entries, `[daemon]` tmux session name, `[approvals]` session approval defaults, `[limits]` tool-call and history caps, webhook, ghost limits) |
| `scripts/` | Executable automation scripts (chmod 700, `.sh` / `.py`) |
| `runbooks/` | Procedure runbooks (markdown + YAML frontmatter) |
| `memory/` | Persistent AI memory (`session/`, `knowledge/`, `incidents/`) |
| `bin/` | Symlinks and wrappers for executables |
| `lib/` | Shared SDK modules (de_sdk, Python helpers) |
| `var/run/daemoneye.sock` | Unix domain socket ŌĆö IPC between CLI and daemon |
| `var/run/schedules.json` | Scheduled job store (atomic writes) |
| `var/run/pane_prefs.json` | Persisted pane-selection preferences (per session) |
| `var/log/events.jsonl` | Structured event log (every tool call, lifecycle event) |
| `var/log/daemon.log` | Daemon process log |
| `var/log/panes/` | Archived background-window scrollback (full, no cap) |
| `var/log/pipe/` | Live pipe-pane capture logs (ephemeral, cleaned at startup) |
| `etc/prompts/` | System prompt TOML files (written by daemon at startup) |
| `var/log/sessions/` | Per-session JSONL conversation history |

The tree is created by `Config::ensure_dirs()`, which is called at the top of `main()` before any subcommand and explicitly at the start of `run_setup()`. The AI agent's `read_file` tool is blocked only from `etc/config.toml` and `etc/prompts/sre.toml` (which contain API credentials). All other paths under `~/.daemoneye/` are readable, including `var/log/`, `var/run/`, `scripts/`, `runbooks/`, and `memory/`. The `edit_file` tool remains fully blocked from all of `~/.daemoneye/` ŌĆö daemoneye-managed data should be written only through dedicated tools.

`daemoneye setup` supports three overwrite flags for post-upgrade refresh:
- `--overwrite-bin` ŌĆö force-copies the current executable to `~/.daemoneye/bin/daemoneye` (skipped by default when a copy already exists).
- `--overwrite-memory` ŌĆö calls `config::overwrite_knowledge_memories()` to force-write all six built-in knowledge memory files; user-created memories in the same directory are unaffected.
- `--overwrite-all` ŌĆö implies both flags above and additionally calls `config::overwrite_sre_prompt()` to refresh `etc/prompts/sre.toml`. User `config.toml` is never auto-overwritten by any flag.

## 3. Core Components

### 2.1 User Terminal Environment

- **User's tmux Session**: DaemonEye does not render its own window or emulate a terminal. It operates entirely within the user's pre-existing terminal emulator via `tmux`.
- **Active Working Pane**: The pane where the user is currently working. When DaemonEye is invoked, it reads the content from this pane and injects approved commands here.
- **AI Agent Pane** (`cli/mod.rs`): A tmux pane running `daemoneye chat` or `daemoneye ask`. On startup the pane is resized to 40% of the window width (minimum 80 columns) using `tmux resize-pane`; `query_pane_width` is then used for accurate header rendering, avoiding a `TIOCGWINSZ` race condition. The chat interface features:
  - Width-adaptive header with session ID; the status bar (bottom row) displays the active model name, session token usage (`Xk / Yk ┬Ę Z%`), and the approval state (dimmed `"approvals: off"` when inactive; bold amber ŌÜĪ when active using count-based segments, e.g. `ŌÜĪapprovals: all ┬Ę scripts: 2 ┬Ę files: 1 ┬Ę Ctrl+C revokes`). The hint uses counts rather than names to stay compact. Progressive fallback drops token/model segments at narrow widths; if the hint itself would overflow it is first truncated with `ŌĆ”` (via `truncate_to_visual` in `render.rs`), then dropped entirely. All border calculations use `visual_len()` (strips ANSI escapes, counts Unicode code points) to handle multi-byte box-drawing characters correctly.
  - ANSI syntax highlighting in fenced code blocks: per-language keyword sets, comment styles, string literals, and numeric literals.
  - Inline markdown rendering: bold, italic, and `code` spans; AI prose is tinted bright-white.
  - Context-budget indicator embedded into the right-hand side of the bottom border of the user input box after each `SessionInfo` response. Shows `turn N ┬Ę Xk / Yk tokens ┬Ę Z% remaining`, where the context window (Yk) is derived from `AiConfig::context_window()` per model. Color-coded by percentage: dim when under 50 % used, yellow at 50ŌĆō74 %, bold red at Ōēź 75 %. Driven by the `UsageUpdate { prompt_tokens }` IPC response sent by the daemon after each completed AI turn; the `prompt_tokens` value is carried across turns as a `&mut u32` parameter so subsequent query boxes display the count from the previous turn.
  - `/clear` in-chat command: generates a new session ID so the next message starts a clean context, prints a dim separator, and updates the header hint.
  - `/model` ŌĆö lists all configured model names, highlighting the session's active one. `/model <name>` switches the session to a named `[models.<name>]` entry and updates the status bar immediately; the daemon validates the name and returns `Response::Error` if unknown.
  - `/pane` ŌĆö lists all targetable panes for the session (reads via `Request::ListPanesForSession`), showing each pane's ID, current command, window name, and whether it is the current foreground target (`Ō¢Ė` marker). `/pane %N` pins a specific pane as the session's foreground target (sends `Request::SetPane`); the daemon validates the ID format, updates `SessionEntry.default_target_pane`, persists to `pane_prefs.json`, and responds with `PaneChanged`; the client prints a dim separator confirming the change. The `[FOREGROUND TARGET]` line on the next AI turn carries the updated ID automatically.
  - `/approvals` ŌĆö prints the current approval state: one row each for regular commands, sudo commands, session-approved scripts (listed by name), session-approved runbooks (listed by name), and session-approved file paths (listed with `~`-relative paths). `/approvals revoke` resets all approval state; `/approvals revoke commands|scripts|runbooks|files` revokes a single class. All forms update the status bar immediately. Approval state also resets on `/clear`, `/prompt`, `/refresh`, and Ctrl+C.
  - `SystemMsg` responses rendered with an amber ŌÜÖ prefix.
  - `ToolResult` responses rendered as a dimmed bordered panel capped at 10 rows with a truncation indicator.
  - **Session-level approval**: Five independent approval scopes are tracked in `SessionApproval { regular: bool, sudo: bool, scripts_all: bool, scripts: HashSet<String>, runbooks_all: bool, runbooks: HashSet<String>, file_edits_all: bool, file_edits: HashSet<String> }`, stored in `run_chat_inner` and threaded into `ask_with_session` as `&mut SessionApproval`. The `*_all` flags are class-wide toggles seeded at session start by `SessionApproval::from_config(&config.approvals)` from the `[approvals]` section of `config.toml`. When a `*_all` flag is `true`, the corresponding class is pre-approved for the session and the `[A]pprove for session` prompt option is suppressed as redundant. Per-name `HashSet` entries are still accumulated via `[A]` when the `*_all` flag is `false`. `from_config()` replaces `SessionApproval::default()` at every call site ŌĆö both at session creation and at reset points (`/clear`, `/prompt`, `/refresh`, Ctrl+C).
    - *Terminal commands*: `ToolCallPrompt` responses show `[Y]es / [A]pprove for session / [N]o` or a redirect message. `daemon::utils::command_has_sudo` (regex `(?:^|[;&|])\s*sudo\b`) classifies each command into the `regular` or `sudo` scope. Once a scope is session-approved, subsequent commands display `Ō£ō auto-approved (session)` without prompting.
    - *Script writes*: `ScriptWritePrompt` first renders an ANSI unified diff via `cli::diff::render_diff` ŌĆö new files show all lines in green with `+` prefixes; modifications show red `-` / green `+` lines with cyan `@@` hunk headers. The prompt then offers `[y]es / [A]pprove for session / [N]o`. Pressing `A` adds the script name to `approval.scripts`; subsequent writes to that name auto-approve (diff still shown, gate skipped).
    - *Runbook writes*: identical pattern to script writes, using `approval.runbooks`.
    - *File operations* (`edit_file` tool): `EditFilePrompt` renders a colored unified diff using the same `cli::diff::render_diff` engine. `[y]es / [A]pprove for session / [N]o` or a redirect message. Pressing `A` adds the path to `approval.file_edits`; subsequent operations on that path auto-approve for the session. The redirect (typed message) path sends `EditFileResponse { approved: false, user_message: Some(text) }`, which the daemon handles identically to `ToolCallResponse` with a typed message ŌĆö aborts the tool chain and injects the text as a new user turn.
    - The status bar hint reflects all active scopes using counts to stay compact: e.g. `ŌÜĪapprovals: all ┬Ę scripts: 1 ┬Ę files: 2 ┬Ę Ctrl+C revokes`. When the hint would still overflow at narrow widths it is truncated with `ŌĆ”` before the full hint is dropped entirely.
    - *Diff infrastructure*: `cli::diff` module wraps the `similar` crate (Myers diff). `render_diff(name, existing_content, new_content) -> Vec<String>` returns pre-coloured lines. New file (`existing_content = None`): all lines green `+`, capped at 80 with a `ŌĆ” (N more lines)` trailer. Modified file: grouped unified diff with 3-line context. Identical content: single "no changes" line. Delete (`new_content = ""`): all lines red `-`.
  - **Interactive line editor**: The chat input runs in raw terminal mode via `set_raw_mode` / `restore_termios` (libc `tcsetattr`). An `AsyncStdin` wrapper (`AsyncFd<StdinRawFd>`) registers fd 0 once with tokio's epoll reactor and serves both the raw editor and cooked-mode tool-approval prompts sequentially. `InputLine` holds the character buffer and cursor; `InputState` wraps it with a navigable `Vec<String>` history. `read_key` parses escape sequences (CSI, SS3) with 30 ms inter-byte timeouts. `render_input_multiline` redraws the word-wrapped input area on every keystroke: the area grows dynamically from 1 to a maximum of 5 rows as the buffer length exceeds the available width, consuming rows from the scroll region upward; `resize_input_area` handles the scroll-region resize (`setup_scroll_region_n` / `draw_input_frame_n`), clearing any rows that transition between scroll-region and input-area roles; `collapse_input_area` restores the 1-row layout before returning so callers always see a clean state. The input frame top border displays `user@host` (via `local_user_host()`, same as the chat history query borders) rather than a static label. SIGWINCH is handled in two places: inside the input loop (while the user is typing) and inside `ask_with_session` during both the spinner phase and the streaming phase. A `StreamResizeDims` struct carries `&mut width`, `&mut height`, `start_time`, `model`, `daemon_up`, and a `has_frame` flag into `ask_with_session`; `apply_stream_resize` re-queries dimensions, erases the old 4-row frame, re-establishes `DECSTBM`, and redraws the input frame and status bar. When `has_frame` is false (the greeting turn, before the frame is first drawn), only dimensions are updated and the caller redraws after streaming completes. Pane-selection prompts (`offer_no_sibling_options`, `pick_sibling_pane`) use `sync_read_line()` which temporarily clears `O_NONBLOCK` on fd 0 before the synchronous read and restores it afterward, preventing the prompt from racing past immediately. Supported: ŌåÉ/ŌåÆ cursor, Ōåæ/Ōåō history, Home/End, Ctrl+A/E/K/U, Backspace, Delete, Ctrl+C (clear line), Ctrl+D (delete-forward or EOF on empty).

### 2.2 IPC Layer

- **Protocol**: Newline-delimited JSON over a Unix Domain Socket at `~/.daemoneye/var/run/daemoneye.sock` (resolved at runtime via `config::default_socket_path()`). The socket lives in the user's own home directory rather than `/tmp` so other local users cannot create a symlink at that path or connect to it.
- **Request types**:
  - `Ping` ŌĆö liveness check.
  - `Shutdown` ŌĆö graceful stop.
  - `Status` ŌĆö query daemon status (uptime, sessions, provider, model, available models, circuit state); daemon responds with `DaemonStatus` (F1).
  - `Ask { query, tmux_pane, session_id, chat_pane, prompt, chat_width, tmux_session, target_pane, model }` ŌĆö start or continue a conversation turn. `tmux_session` and `target_pane` are resolved client-side before connecting; `target_pane` is stored as `SessionEntry.default_target_pane` so `find_best_target_pane()` never prompts mid-conversation. `model` is an optional name that pins the session to a specific `[models.<name>]` entry on the first turn.
  - `Refresh` ŌĆö re-collect system context (OS, memory, processes, history) via `refresh_sys_context()`; daemon responds with `Ok`.
  - `SetModel { session_id, model }` ŌĆö switch the active model for a session. The daemon validates the name against configured `[models.*]` keys and responds with `ModelChanged` on success or `Error` if unknown.
  - `ListModels { session_id }` ŌĆö list all configured model names and the session's current active model; daemon responds with `ModelList`.
  - `SetPane { session_id, pane_id }` ŌĆö pin the foreground target pane for a session. The daemon validates the pane ID format, updates `SessionEntry.default_target_pane`, persists to `pane_prefs.json`, and responds with `PaneChanged`. Used by the `/pane %N` slash command.
  - `ListPanesForSession { session_id }` ŌĆö list all targetable panes known to the daemon for a session (excludes the chat pane and daemon-owned background windows); daemon responds with `PaneList`. Used by the `/pane` slash command.
  - `ToolCallResponse { id, approved, user_message }` ŌĆö user's approval decision for a proposed command. When `approved=false` and `user_message` is `Some`, the daemon aborts the pending tool chain and injects the text as a new user turn so the AI can course-correct.
  - `CredentialResponse { id, credential }` ŌĆö user-supplied sudo password for an approved background command, sent in response to `CredentialPrompt`.
  - `PaneSelectResponse { id, pane_id }` ŌĆö user's pane selection, sent in response to `PaneSelectPrompt`.
  - `ScriptWriteResponse { id, approved }` ŌĆö user's approval decision for a proposed script write, sent in response to `ScriptWritePrompt`.
  - `ScheduleWriteResponse { id, approved }` ŌĆö user's approval decision for a proposed scheduled job, sent in response to `ScheduleWritePrompt`.
  - `RunbookWriteResponse { id, approved }` ŌĆö user's approval decision for a proposed runbook write, sent in response to `RunbookWritePrompt`.
  - `RunbookDeleteResponse { id, approved }` ŌĆö user's approval decision for a proposed runbook deletion, sent in response to `RunbookDeletePrompt`.
  - `EditFileResponse { id, approved, user_message }` ŌĆö user's approval decision for a proposed file operation (`edit_file` tool), sent in response to `EditFilePrompt`. `user_message` carries a redirect message when the user types instead of y/n, handled identically to `ToolCallResponse` with a typed message.
  - `NotifyActivity { pane_id, hook_index, session_name }` ŌĆö background pane activity from a tmux hook (alert-activity or pane-died); broadcasts `pane_id` on `BG_DONE_TX`. `pane_id` is validated via `is_valid_pane_id()` before use (S11).
  - `NotifyComplete { pane_id, exit_code, session_name }` ŌĆö background command finished; broadcasts `(pane_id, exit_code)` on `COMPLETE_TX` (primary completion path for background commands).
  - `NotifyFocus { pane_id, session_name }` ŌĆö signals that a pane received focus (`pane-focus-in` hook, N1); daemon calls `cache.set_active_pane(pane_id)` to update the active-pane state immediately without waiting for the next 2 s poll.
  - `NotifyWindowChanged { session_name }` ŌĆö signals that the active window changed (`session-window-changed` hook, N2); daemon calls `cache.refresh_windows()` to refresh window topology immediately.
  - `NotifySessionCreated { session_name }` ŌĆö signals that a new tmux session was created (`after-new-session` global hook, N14); daemon calls `install_session_hooks()` for the new session so all per-session monitoring hooks are registered automatically.
  - `NotifySessionClosed { session_name }` ŌĆö signals that a tmux session was destroyed (`session-closed` per-session hook, A6); daemon calls `cleanup_bg_windows()` on all matching `SessionEntry` records and removes them from the session store. If `session_name` matches the `managed_session` configured at startup, the daemon immediately recreates the session (`tmux new-session -d -s <name>`), repopulates `bg_session` and the cache, and logs `INFO "Recreated managed tmux session 'ŌĆ”' after close."`; the `after-new-session` hook then reinstalls per-session monitoring hooks automatically.
  - `NotifyClientAttached { session_name }` ŌĆö signals that a terminal client attached to a tmux session (`client-attached` global hook, N15); daemon clears `last_detach` on all `SessionEntry` records whose `tmux_session` matches, suppressing any pending catch-up brief.
  - `NotifyClientDetached { session_name }` ŌĆö signals that the last terminal client detached from a tmux session (`client-detached` global hook, N15); daemon records `last_detach = now` and `messages_at_detach = messages.len()` on matching session entries so the next `Ask` can generate a catch-up brief.
  - `NotifyResize { width, height, session_name }` ŌĆö signals that the attached terminal was resized (`client-resized` per-session hook, N8); daemon calls `cache.set_client_size(width, height)` so the `[CLIENT VIEWPORT]` block stays current.
- **Response types**:
  - `Ok` ŌĆö signals successful completion of a turn.
  - `Error(String)` ŌĆö error from the daemon or AI provider.
  - `SessionInfo { message_count }` ŌĆö sent once before streaming; carries prior turn count.
  - `Token(String)` ŌĆö streaming AI response token.
  - `SystemMsg(String)` ŌĆö daemon notification (sudo alert, pane switch, approval timeout, catch-up brief, etc.). Rendered with an amber ŌÜÖ prefix.
  - `ToolCallPrompt { id, command, background, target_pane }` ŌĆö daemon requests user approval for a command. `target_pane` carries the best-guess pane ID for foreground commands so the client can resolve the window-relative index and visually highlight the target pane during the approval window.
  - `CredentialPrompt { id, prompt }` ŌĆö daemon requests the sudo password for an approved background command.
  - `PaneSelectPrompt { id, panes: Vec<PaneInfo> }` ŌĆö daemon cannot determine target pane; client displays the list and returns `PaneSelectResponse`.
  - `ToolResult(String)` ŌĆö captured output of an approved command, sent before the AI continues. Rendered as a dimmed bordered panel (capped at 10 rows).
  - `ScriptWritePrompt { id, script_name, content }` ŌĆö daemon shows full script content to user before writing; requires `ScriptWriteResponse`.
  - `ScheduleWritePrompt { id, name, kind, action }` ŌĆö daemon shows scheduled job details before creation; requires `ScheduleWriteResponse`.
  - `RunbookWritePrompt { id, runbook_name, content }` ŌĆö daemon shows full runbook content to user before writing; requires `RunbookWriteResponse`.
  - `RunbookDeletePrompt { id, runbook_name, active_jobs }` ŌĆö daemon warns if active scheduled jobs reference the runbook before deletion; requires `RunbookDeleteResponse`.
  - `EditFilePrompt { id, path, operation, existing_content, new_content, dest_path }` ŌĆö daemon requests user approval for a file operation. `operation` is `"edit"` | `"create"` | `"delete"` | `"copy"`. `existing_content` / `new_content` carry the before/after file content so the client can call `render_diff()` without any filesystem access. `dest_path` is set for `"copy"` operations. Requires `EditFileResponse`.
  - `ScheduleList { jobs: Vec<ScheduleListItem> }` ŌĆö list of scheduled jobs with id, name, kind, action, status, last_run, next_run.
  - `ScriptList { scripts: Vec<ScriptListItem> }` ŌĆö list of scripts in `~/.daemoneye/scripts/` with name and size.
  - `RunbookList { runbooks: Vec<RunbookListItem> }` ŌĆö list of runbooks in `~/.daemoneye/runbooks/` with name and tags.
  - `MemoryList { entries: Vec<MemoryListItem> }` ŌĆö list of memory entries with category, key, and optional summary.
  - `UsageUpdate { prompt_tokens: u32 }` ŌĆö sent once per completed AI turn immediately before `Ok`; carries the prompt token count so the CLI can display an accurate context-budget indicator.
  - `ModelChanged { model }` ŌĆö confirmation that the session's active model was switched (response to `SetModel`).
  - `ModelList { models: Vec<String>, active: String }` ŌĆö sorted list of all configured model names and the session's current active model (response to `ListModels`).
  - `PaneChanged { pane_id, description }` ŌĆö confirmation that the session's foreground target pane was pinned (response to `SetPane`). `description` is a human-readable label (`%7 (bash)`).
  - `PaneList { panes: Vec<(String, String, String, bool)> }` ŌĆö list of targetable panes `(pane_id, current_cmd, window_name, is_current_target)` (response to `ListPanesForSession`).
  - `DaemonStatus { uptime_secs, pid, active_sessions, provider, model, available_models, socket_path, schedule_count, circuit_state, redaction_counts }` ŌĆö response to `Request::Status` (F1). `available_models` is a sorted list of all `[models.*]` names from config. `redaction_counts` is a `HashMap<String, usize>` of per-type redaction totals since daemon start (keys: `"AWS Key"`, `"Private Key"`, `"GCP Key"`, `"JWT"`, `"GitHub Token"`, `"DB URL"`, `"Secret"`, `"Card"`, `"SSN"`, `"User Defined"`); all built-in types are always present even when their count is zero; `"User Defined"` accumulates hits from any `[masking] extra_patterns` the user has configured. Old daemons that omit fields are handled via `#[serde(default)]`.
- Each client connection handles exactly one request/response lifecycle. `Ask` connections receive a token stream (interleaved with zero or more tool-call approval round-trips) terminated by `Ok` or `Error`. Incoming IPC messages are capped at 1 MiB (`MAX_IPC_MESSAGE_BYTES` in `server.rs`); oversized payloads are rejected with `Response::Error` and the connection is closed (S14).

### 2.3 DaemonEye Background Daemon

- **AI Agent Engine** (`daemon/server.rs` & `daemon/executor.rs`): Orchestrates the full request lifecycle. `server.rs` handles the IPC connection loop, loads session history, builds the per-turn prompt, and streams AI events. `executor.rs` processes the resulting tool calls, handling user approval gates and dispatching background/foreground execution. Approval gate logic (send prompt ŌåÆ await response ŌåÆ timeout/deny/approve ŌåÆ log) is centralized in `prompt_and_await_approval()`, used by both the foreground and background arms. On approval timeout, a `Response::SystemMsg` is sent to the client before returning the tool result so the user sees a clear notification even if their approval window had already closed (A3). A per-turn loop guard enforces two-tier tool-call caps: approval-gated tools (`run_terminal_command`, `edit_file`, `write_script`, `write_runbook`, `schedule_command`, `spawn_ghost_shell`, `delete_script`, `delete_runbook`, `delete_schedule`) are capped at 2 calls per turn; all no-approval tools (memory CRUD, reads, search, etc.) are capped at 30 per turn; total cap across all tools is 50 per turn. Calls blocked by the cap receive an error result rather than being silently dropped.

  **Prompt construction is split by turn:**
  - *First turn*: Full host context (`## Host Context`, `## Execution Context`, session memory block) + fresh terminal snapshot from `cache.get_labeled_context()` (`## Terminal Session`). `session_summary` is computed only here.
  - *Subsequent turns*: Token-budget note only (`[Token Budget] Context at Xk / Yk tokens (Z% used) ŌĆ”`) + the user's query. No terminal snapshot is included ŌĆö the AI calls `get_terminal_context` when it needs one.

  After every AI turn `server.rs` sends `Response::UsageUpdate { prompt_tokens }` to the client and persists the value in `SessionEntry.last_prompt_tokens` so budget warnings in the next turn's prompt are accurate. `SessionEntry` also carries three N15 fields: `tmux_session: String` (the tmux session name from the most recent `Ask`, used to correlate IPC hook notifications with the correct entry), `last_detach: Option<Instant>` (set by `NotifyClientDetached`, cleared by `NotifyClientAttached`), and `messages_at_detach: usize` (history length at the moment of detach, used to identify new event messages for the catch-up brief).
- **Execution Context Detector** (`daemon/session.rs`): On the first turn of each session, detects the daemon's local hostname (via `/proc/sys/kernel/hostname`) and whether the user's tmux pane is running `ssh` or `mosh` (via `tmux display-message #{pane_current_command}`). Injects a structured `## Execution Context` block into the prompt so the AI knows which machine each execution mode targets.
- **Dual Command Execution** (`daemon/executor.rs` & `daemon/background.rs`):
  - *Background mode* (`background=true`): Handled by `daemon/background.rs`. Runs in a dedicated tmux window on the daemon host. Window names follow the format `de-bg-<pane_num>-<unix_ts>-<cmd_slug>` (e.g. `de-bg-42-1712937600-cargo-build`): the window is first created with a temporary name so that `create_job_window` can return the pane ID, then immediately renamed via `tmux rename-window`. `pane_num` is the pane ID with `%` stripped; `unix_ts` is a 10-digit Unix epoch; `cmd_slug` is produced by `sanitize_cmd_for_window(cmd, 30)` which strips wrappers (`sudo`, `env`, `nohup`, env-var assignments), takes the basename, skips to the script argument for interpreter invocations (`bash`, `python3`, `node`, etc.), replaces non-alphanumeric characters with `-`, collapses runs of `-`, and truncates to 30 chars. Ghost shell windows use the same logic with their own prefixes (`de-gs-bg-*`, `de-gs-sj-*`). The shell is detected via `pane_current_command` immediately after window creation to select the correct exit-code variable (`$?` for POSIX shells; `$status` for fish/csh/tcsh). The command is wrapped as `cmd; __de_ec=$?; daemoneye notify complete <pane_id> $__de_ec <session>` (fish: `set __de_ec $status`) ŌĆö **no** `; exit` ŌĆö so the shell stays alive after the command finishes, keeping the window available for follow-up commands. `run_background_in_window` returns immediately with the pane ID; a background tokio task monitors completion via two paths: (primary) the command wrapper calls `daemoneye notify complete` via IPC, which broadcasts `(pane_id, exit_code)` on `COMPLETE_TX` ŌĆö window persists for follow-up commands, `exit_code` updated in `bg_windows`; (fallback) the shell itself crashes or exits, `pane-died` hook fires and broadcasts the pane ID on `BG_DONE_TX`, exit code recovered via `pane_dead_status` ŌĆö window is GC-killed and entry removed from `bg_windows`. On either path the pane scrollback is archived to `~/.daemoneye/var/log/panes/<win_name>.log` and a `[Background Task Completed]` `Message` is injected into the session history with the exit code, masked output, a reminder to call `close_background_window(pane_id)` when done, and whether the pane is still open. All windows are killed by `cleanup_bg_windows()` when the session expires. The AI can explicitly close a background window via the `close_background_window(pane_id)` tool (no approval required), which calls `tmux kill-window` and removes the entry from `bg_windows`. A dedicated `bg-window-gc` supervised task (60 s cycle) calls `gc_bg_windows()`, which queries all live panes via a single `list_panes_detailed()` call and kills: (a) windows whose pane no longer exists in tmux (`pane_gone`); (b) dead panes (remain-on-exit) with `last_activity` older than 30 s (`pane_dead`); (c) windows with `exit_code.is_some()` whose shell has been idle for more than 120 s (`idle_completed`). The GC also performs an orphan sweep: daemon-prefixed windows present in tmux but not tracked by any session are killed with reason `"orphan"` ŌĆö handling windows left behind after a daemon restart or a missed completion signal. The GC decision logic is factored into the pure `plan_gc_actions(windows, pane_info, now_unix)` function for unit-testability. If the command contains `sudo`, the daemon first runs `sudo -n true` as a non-interactive credential-cache probe (`sudo_credentials_cached()` in `daemon/utils.rs`); if credentials are cached the command proceeds without any password interaction. Otherwise a `CredentialPrompt` IPC response is sent; the client reads the password with echo disabled in the chat pane and returns it via `CredentialResponse`. Before injection the command is prefixed with `SUDO_PROMPT='[de-sudo-prompt]'` so that prompt detection is locale-independent (P5). The credential is injected synchronously (before `run_background_in_window` returns) after `wait_for_sudo_prompt_and_inject()` detects the sentinel or standard `[sudo]`/`password` strings in the pane scrollback. `sudo_auth_failed()` then polls for "Sorry, try again." ŌĆö if detected, the failure is logged at WARN; the command's non-zero exit code reaches the AI via the normal `[Background Task Completed]` completion path. The credential is wrapped in `zeroize::Zeroizing<String>` (which zeroes the memory on drop), and the raw JSON credential line is also explicitly zeroized immediately after parsing. The credential is never logged or transmitted to the AI.
  - *Foreground mode* (`background=false`): Injects the command into the user's active tmux pane via `tmux send-keys`. A temporary `pane-title-changed[@de_fg_N]` hook is attached to the target pane before injection so activity signals are available to all three completion-detection paths. Hook cleanup is managed by `FgHookGuard` (a RAII struct in `executor.rs` with a `Drop` impl that calls `tmux set-hook -u` for all registered hooks), ensuring hooks are removed on any exit path including early returns via `?` and send-keys failures. Immediately after `send_keys` succeeds the target pane is highlighted via `tmux select-pane -P bg=colour17` (dark-blue tint) so the user can see which pane the agent is working in; a second `select-pane -t <chat_pane>` is issued immediately after each style change so the user's active pane is never displaced. The highlight is removed via `select-pane -P default` just before `capture_pane` at command completion; the client removes it immediately on denial so no highlight lingers on an unapproved command. Completion is detected via a three-way branch in `executor.rs`: (1) **Interactive commands** (`ssh`, `mosh`, `telnet`, `screen`) ŌĆö detected by `is_interactive_command()` (`utils.rs`). The daemon polls for a shell-prompt pattern (`looks_like_shell_prompt`) up to 10 s, then returns immediately with an `[Interactive session started]` tool result containing the destination host and instructions to use `run_terminal_command(target_pane=...)` for follow-up commands. Non-interactive sub-cases (`ssh host command`, `ssh -N`) bypass this path. (2) **Remote panes** (foreground process is `ssh`/`mosh-client`) ŌĆö output-stability polling: two consecutive identical 10-line snapshots at 500 ms intervals, 30 s deadline. (3) **Local panes** ŌĆö event-driven and PID-based: before injecting the command, installs an `alert-silence[@de_fg_N]` hook and sets `monitor-silence 2` on the target window (N9); this causes tmux to fire the same `notify activity` IPC signal after 2 s of pane silence, acting as a secondary completion signal. Completion is determined by PID comparison: `pane_pid(target_str)` is called once before `send_keys` to record the shell PID (`idle_pid`). The "child started" window polls `pane_pid` until it differs from `idle_pid`; the subsequent completion wait loop (hook-driven + 500 ms slow-poll fallback) breaks when `pane_pid` returns to `idle_pid`. Using the PID rather than the process name makes completion detection unambiguous for commands that share the shell name ŌĆö e.g. `bash script.sh` or `bash -c "..."` where both shell and child report as `bash`. When `pane_pid` is unavailable (returns 0), the child-started window exits immediately (`saw_child = true`) and the completion loop relies solely on hook signals. 45 s deadline. Both the `alert-silence` hook and `monitor-silence` are removed after the wait loop exits. For sudo commands on **local panes**, `sudo_credentials_cached()` runs `sudo -n true` immediately after `send_keys` ŌĆö if credentials are cached the password workflow is skipped entirely; the cached-credentials fast path is confirmed by checking that `pane_pid` returns to `idle_pid` (i.e. sudo has already exited) rather than matching command names. If not cached, a `CredentialPrompt`/`CredentialResponse` IPC round-trip collects the password from the user in the chat pane (echo disabled, no pane focus switch); `wait_for_sudo_prompt_and_inject()` then polls `capture_pane` for a sudo password prompt and injects the credential via `send_keys`. Authentication failures ("Sorry, try again.") are detected by `sudo_auth_failed()` and trigger re-prompts up to `MAX_SUDO_RETRIES` (3) times; if all attempts fail or the user times out, `FgHookGuard` is dropped, the pane highlight is cleared, and the daemon returns a structured `ToolCallOutcome::Result` error to the AI naming the failure and suggesting `daemoneye install-sudoers`. For **remote panes**, `pane_current_command` polling (100 ms intervals, 3 s window) is retained for the credential cache check since the credential cache lives on the remote host and `pane_pid` reflects the remote shell. **Exit code capture**: after the post-command capture delay, `read_pane_exit_status(pane_id)` (`tmux/pane.rs`) reads the tmux session environment key `DE_EXIT_<N>` (e.g. `DE_EXIT_3` for pane `%3`) and calls `daemon::stats::finish_command(cmd_id, exit_code)` to record the outcome and increment `COMMANDS_FG_SUCCEEDED` / `COMMANDS_FG_FAILED`. This key is written by an optional shell hook (`PROMPT_COMMAND` in bash, `precmd_functions` in zsh) that runs automatically after every command completes. When the hook is not configured, `finish_command` is called with exit code 0 (optimistic success). `edit_file` calls (all operations, both local and remote paths) use the deterministic operation result instead of the shell hook. The `edit_file` tool now supports four operations routed in `daemon/executor/file_ops.rs`: `"edit"` (atomic `old_string ŌåÆ new_string` replacement, full-file diff sent to client), `"create"` (new file from `content`; fails if file exists; new-file green diff), `"delete"` (reads file then removes it; all-red deletion diff), and `"copy"` (duplicates `path` to `dest_path`; fails if dest exists; shows dest content as green new-file diff). All four operations send an `EditFilePrompt` IPC response and await an `EditFileResponse`; the shared `await_edit_file_response()` helper handles the timeout, EOF, and user-message-redirect paths. Remote operations use hex-encoded Python3/Perl scripts for `edit` and `create`; `delete` and `copy` use shell `rm`/`cp` commands with `DE_OK:`/`DE_ERROR:` markers. Event names logged to `events.jsonl`: `file_edit`, `file_create`, `file_delete`, `file_copy`.
- **Passive Pane Monitoring** (`daemon/server.rs` & `daemon/background.rs`): The daemon installs four global hooks at startup: `pane-died` (existing), `after-new-session` (N14: automatically calls `install_session_hooks()` for any tmux session created after the daemon starts, so new sessions get monitoring hooks without manual reconfiguration), `client-attached` (N15: fires when a terminal client re-attaches to a tmux session ŌĆö daemon clears `last_detach` on matching `SessionEntry` records, suppressing any pending catch-up brief), and `client-detached` (N15: fires when the last terminal client detaches ŌĆö daemon records `last_detach = now` and `messages_at_detach = messages.len()` on matching `SessionEntry` records so that the next `Ask` request can generate a catch-up brief). Failure to register `pane-died` is always logged at ERROR. Failures for `after-new-session`, `client-attached`, and `client-detached` are also logged at ERROR (not WARN) because these hooks are load-bearing for automatic session monitoring and catch-up briefs; a silent failure at this level would leave core functionality broken with no diagnostic trail. The catch-up brief is generated in `server.rs` at the start of each `Ask` turn: if `away_secs >= 30` and new event messages have appeared in history since detach (identified by prefixes `[Background Task Completed`, `[Webhook Alert]`, `[Watchdog]`, `[Watch Pane`, `[Ghost Shell Started]`, `[Ghost Shell Completed]`, `[Ghost Shell Failed]`), the daemon sends a `Response::SystemMsg` with a `[Catch-up] N events while you were away (Xm): ŌĆ”` summary immediately after `Response::SessionInfo`. `install_session_hooks()` registers four per-session hooks via `tmux set-hook`: `alert-bell` (existing background-completion fallback), `pane-focus-in` (N1: fires whenever the user moves focus to a different pane ŌĆö notifies the daemon via `NotifyFocus` IPC, enabling `cache.set_active_pane()` to update the active-pane state instantly rather than waiting up to 2 s for the next poll), `session-window-changed` (N2: fires whenever the active window changes ŌĆö notifies the daemon via `NotifyWindowChanged` IPC, triggering `cache.refresh_windows()` for an immediate topology update), and `session-closed` (A6: fires when the tmux session is destroyed ŌĆö sends `NotifySessionClosed` IPC, which triggers `cleanup_bg_windows()` on all matching `SessionEntry` records and removes them from the session store; the session name is embedded directly in the hook command rather than via `#{session_name}` format expansion since the session may already be gone when the hook fires; per-session hooks registered with `-t` are automatically removed by tmux on session destruction so no explicit hook teardown is needed). All hooks use `#{pane_id}` as a tmux format variable so they fire for every pane in the session. Session names are shell-escaped via `shell_escape_arg()` (`daemon/utils.rs`) before embedding in the `run-shell` hook string. The function handles `\`, `"`, `$`, `` ` `` (shell-level) and `'` (tmux-level: replaced with `'\''` ŌĆö end-single-quote, backslash-escaped quote, begin-single-quote ŌĆö so session names with literal single-quotes do not break the outer `'ŌĆ”'` tmux command context). The hook command is `run-shell -b '<binary> notify activity #{pane_id} 0 "<session>"'` using the absolute path of the running binary. The `NotifyActivity`, `NotifyComplete`, and `NotifyFocus` IPC handlers each validate their `pane_id` argument via `is_valid_pane_id()` (defined in `server.rs`: must start with `%` and be followed by one or more ASCII digits) before taking any action ŌĆö invalid IDs are logged at WARN level and discarded. The `NotifyActivity` IPC handler (in `server.rs`) broadcasts the pane ID on `BG_DONE_TX`, waking the background completion monitor's fallback path. The `NotifyComplete` IPC handler broadcasts `(pane_id, exit_code)` on `COMPLETE_TX`, waking the primary completion path. For `watch_pane`, a temporary `pane-title-changed[@de_wp_N]` hook is installed per invocation; the spawned monitor task races the hook signal against a 500 ms `pane_current_command` poll ŌĆö when the foreground process returns to a known shell name, the command is considered done; a `[Watch Pane Complete]` or `[Watch Pane Timeout]` message is injected into the session history and a `tmux display-message` overlay is shown. For background job completion, the monitoring task in `run_background_in_window` selects between `COMPLETE_TX` (primary, carries exit code directly) and `BG_DONE_TX` (fallback, exit code recovered via `pane_dead_status`), then handles capture, archival, `[Background Task Completed]` session history injection, `tmux display-message` status banner, and GC window cleanup (fallback path) or window persistence (primary path). `notify_job_completion()` (in `background.rs`) is used exclusively by scheduled/watchdog jobs.
- **Event Logger** (`daemon/utils.rs`): Appends structured JSON records to `~/.daemoneye/var/log/events.jsonl` for every tool call (approved, denied, timed out), lifecycle event, and AI interaction.
- **Multi-Turn Session Store**: An in-memory `HashMap<session_id, SessionEntry>` shared across connections. Entries are pruned after 30 minutes of inactivity. History is bounded to 40 messages to keep context windows manageable. `SessionEntry` includes an `is_ghost: bool` flag, `ghost_config: Option<GhostConfig>` inherited from the triggering runbook, and `started_at: DateTime<Utc>` ŌĆö a wall-clock timestamp set when the session is created, used by the session digest to filter events and detect artifacts created during the session's lifetime. Session files are written append-only via `append_session_message()` on the hot path; `write_session_file()` (full rewrite) is reserved for post-compaction rewrites. `write_session_file` writes atomically: tmp file ŌåÆ `sync_all()` ŌåÆ rename, so a crash mid-write leaves the old on-disk history intact. Both functions log WARN on I/O failure rather than silently swallowing errors.

  **Session Digest** (`daemon/digest.rs`): Compaction is driven by prompt-token pressure ŌĆö at `DIGEST_PCT` (60%) of the active model's context window (or when the `limits.max_history` cap is hit, default 80) the daemon runs a structured digest-and-compact pass. `DIGEST_THRESHOLD` (20 messages) is the minimum floor that must be crossed before either path fires. The digest is *hybrid* when `[digest].narrative_enabled = true` in `config.toml` (off by default): `build_narrative_summary()` calls a cheap model (the optional `[models.digest]` config entry, falling back to `[models.default]`) with `use_tools=false` to turn the about-to-be-dropped turns into a short natural-language narrative capturing causal threads, decisions, and unresolved state that raw event counts cannot. The narrative call runs with a 20 s timeout; on timeout or backend error the structured digest still fires (best-effort). When the toggle is off the narrative step is skipped entirely ŌĆö no extra API call is made. `build_session_digest()` then prepends the narrative (when available) to the deterministic tally: `events.jsonl` filtered by session ID and `started_at` for commands (success/fail with names), files edited, cumulative token usage, background window lifecycle, webhook alerts, and ghost shell activity; plus a filesystem scan for artifacts created or modified since session start (runbooks, scripts, memories, scheduled jobs). The result is formatted as a `[Session Digest ŌĆö N messages compacted]` block. `compact_with_digest()` replaces old messages with `[first_message, digest_as_assistant, last 16 messages]`, snapping the tail forward via `next_clean_turn_start()` so the cut never lands between an assistant's tool-calls and the following user's tool-results (compaction skips rather than risk an orphaned pair). `planned_tail_start()` exposes the same cut point so the server can compute which slice to hand to the narrative step before invoking compaction. At `ELISION_PCT` (50%) ŌĆö below the digest threshold ŌĆö `elide_old_tool_results()` instead replaces oversized tool outputs (>3000 chars) in older messages with `[elided: tool X produced N chars]` placeholders, preserving turn structure while freeing stale bytes. The digest logs four structured events to `events.jsonl` for its own workflow: `session_digest_start`, `session_digest_events_scanned` (tally counts), `session_digest_artifacts_found` (counts by type), and `session_digest_complete` (digest size, artifact count).

- **Autonomous Ghost Shells** (`daemon/ghost.rs` & `daemon/policy.rs`): A specialized session type triggered by webhooks for unattended alert response. Ghost shell background command windows use the `de-gs-bg-*` and `de-gs-sj-*` prefixes (same `pane_num-unix_ts-cmd_slug` naming convention) ŌĆö all prefix constants are defined in `daemon/mod.rs`. The `INCIDENT_WINDOW_PREFIX` (`de-gs-ir-`) is reserved for future main-session incident-response windows and is used in cache filtering/classification today. These windows are visible in the AI context as `[BACKGROUND PANE ŌĆ” [ghost]]` (classified as background by `get_labeled_context()` and annotated `[ghost]`) and in `list_panes` output with a `[ghost]` tag, so any AI agent ŌĆö including a ghost shell itself ŌĆö can discover and target incident panes. Ghost shells operate under an OS-delegation trust model enforced by `GhostPolicy`: non-sudo commands always run freely (the daemon runs as the same user, so OS file permissions are the security boundary); sudo commands are only allowed when the script basename is listed in the runbook's `auto_approve_scripts` frontmatter field AND a NOPASSWD sudoers rule has been installed via `daemoneye install-sudoers`. The optional `auto_approve_commands: bool` flag (runbook frontmatter or daemon-wide via `[approvals] ghost_commands = true` in `config.toml`) does not change the permission boundary ŌĆö it merely makes the AI's ability to run non-sudo commands explicit in the ghost system prompt, preventing the model from unnecessarily withholding investigation commands. The two sources are OR-merged in `GhostManager::start_session()`: `ghost_config.auto_approve_commands |= ghost_commands_default`.
  - **Detection signal**: `watchdog_system_prompt()` (`runbook.rs`) instructs the watchdog model to emit `GHOST_TRIGGER: YES` or `GHOST_TRIGGER: NO` on the final line of its response. `parse_ghost_trigger()` in `webhook.rs` scans lines in reverse, case-insensitive; falls back to a legacy `ALERT` keyword check when the structured field is absent. The webhook analysis call uses `use_tools=false` so the model is forced to generate plain text rather than tool calls (Gemini receives `toolConfig.functionCallingConfig.mode="NONE"`; Anthropic and OpenAI omit the tool list). The outcome is logged at INFO level with a structured `trigger_reason` field (`"GHOST_TRIGGER: YES"`, `"GHOST_TRIGGER: NO"`, `"legacy ALERT keyword"`, `"no GHOST_TRIGGER line and no ALERT keyword"`, `"empty_response ŌĆö model returned no tokens"`, or `"api_error ŌĆö response empty"`).
  - **Turn loop**: `trigger_ghost_turn()` in `server.rs` drives the ghost AI loop. Each iteration creates a **fresh** `(ai_tx, ai_rx)` channel ŌĆö the sender is moved (not cloned) into the spawned task so the channel closes when the task exits and `recv()` unblocks naturally. A `timeout_at` guard caps each turn. The initial history entry must use `role: "user"` (Anthropic API rejects assistant-first conversations). `execute_tool_call()` is called with `tokio::io::sink()` as the writer and `BufReader::new(tokio::io::empty())` as the reader ŌĆö writes are discarded immediately (never block) and reads return EOF at once. Using a bounded `duplex` buffer for this would deadlock once accumulated `send_response_split()` writes from tool results exceed the buffer capacity, leaving the job stuck as `Running` forever.
  - **Turn budget**: Two-level budget. The runbook frontmatter field `max_ghost_turns: N` (parsed by `parse_usize_field()`; `0` = use daemon default) is clamped to `config.ghost.max_ghost_turns` ŌĆö the daemon-wide hard ceiling configured in `config.toml` under `[ghost]`, defaulting to 20. This prevents runbooks from granting themselves more turns than the operator allows. Enforced in `trigger_ghost_turn`; a WARN is logged when the limit is hit. The ghost system prompt includes a `Turn Budget: N (hard limit)` line so the model can self-regulate.
  - **run_with_sudo**: `GhostConfig` carries a `run_with_sudo: bool` field (parsed from runbook frontmatter `run_with_sudo: true/false`, default `false`). `GhostPolicy::resolve_command()` prepends `sudo` to the full script path when `run_with_sudo=true`. `GhostPolicy::is_safe()` strips a leading `sudo` token before basename extraction so whitelisted scripts are still recognised when the AI emits them with or without the `sudo` prefix. The ghost system prompt annotates pre-approved scripts with "(executed with sudo)" when the flag is set.
  - **Auto-denial logging**: When `is_safe()` returns `false` (a sudo command not in the whitelist), the denial is logged at INFO with the full command and whitelist contents so operators can diagnose mismatches without reading runbook source.
  - **Session validation**: `session_exists()` (`tmux/session.rs`, wraps `tmux has-session -t`) is called before the AI loop to guard against stale session names.
  - **Lifecycle notifications**: `inject_ghost_event()` in `webhook.rs` injects `[Ghost Shell Started]`, `[Ghost Shell Completed]`, or `[Ghost Shell Failed]` messages into all active sessions and displays them via `tmux display-message` in all chat panes. Each message includes `ŌĆö session log: <path>` (the full path to the session JSONL file, derived from `session_file(session_id)`) so the AI can immediately call `read_file(<path>)` to review the ghost's full conversation history. These prefixes are also scanned by `build_catchup_brief()` (see catch-up brief above).
  - **Active count**: `GHOSTS_ACTIVE` atomic in `daemon/stats.rs` is incremented by `inc_ghosts_launched()` and decremented (saturating) by both `inc_ghosts_completed()` and `inc_ghosts_failed()`. Exposed in `Response::DaemonStatus` as `ghosts_active` and shown in `daemoneye status`.
- **tmux Interoperability Layer** (`tmux/`): Executes `tmux capture-pane`, `tmux send-keys`, `tmux list-panes`, `tmux resize-pane`, and `tmux display-message` to read and write to the user's terminal. Key query functions: `pane_current_command` (foreground process name), `pane_pid` (PID of the foreground process ŌĆö shell PID when idle, child PID when a command is running; used for local-pane completion detection), `query_pane_width` / `query_window_width` (column counts for accurate header rendering), `resize_pane_width` (auto-resize chat pane to 40% of window width on startup), `list_panes_detailed()` (single tab-delimited `list-panes -a -F` call that fetches 16 fields ŌĆö session name [0], window name [1], pane ID [2], command [3], path [4], title [5], dead [6], dead_status [7], scroll position [8], history size [9], copy-mode flag [10], synchronized flag [11], `pane_activity` Unix timestamp [12], `pane_start_command` [13], `pane_index` [14], and `pane_pid` [15] ŌĆö for every pane in one subprocess via `splitn(16)`; field [12] is parsed as `u64` into `RichPaneInfo.last_activity`; field [13] is stored as `RichPaneInfo.start_cmd` (N5) and surfaced in context and `list_panes` output when it differs from `current_cmd`; field [15] is stored as `RichPaneInfo.pane_pid: u32`), `session_environment()` (calls `tmux show-environment` and filters the output against a 20-key allowlist ŌĆö AWS_PROFILE, AWS_DEFAULT_REGION, AWS_REGION, KUBECONFIG, KUBE_CONTEXT, KUBECTL_CONTEXT, VAULT_ADDR, DOCKER_HOST, DOCKER_CONTEXT, ENVIRONMENT, APP_ENV, NODE_ENV, RAILS_ENV, RACK_ENV, VIRTUAL_ENV, CONDA_DEFAULT_ENV, GOPATH, GOENV, JAVA_HOME, LANG/LC_ALL ŌĆö returning a keyŌåÆvalue map), `pane_dead_status()` (queries `#{pane_dead}\t#{pane_dead_status}` via `display-message` for a specific pane; returns 124 when the pane status is also unavailable, following the POSIX timeout-exit-code convention), `capture_pane_with_escapes()` / `capture_pane_at_scroll_with_escapes()` (R2 variants of the standard capture functions that add `-e` to preserve ANSI SGR colour sequences; output is then passed to `annotate_ansi()` before being included in context).
- **Pipe-Pane Log** (`tmux/pane.rs`, R1): When the first `Ask` request arrives with a source pane that is **not** the chat pane itself, the daemon calls `start_pipe_pane(pane_id)`, which runs `tmux pipe-pane -O -t PANE "cat >> /tmp/de-pipe-N.log"`. If `start_pipe_pane` fails, or if the source pane equals the chat pane (which happens when no sibling target pane has been resolved and the client falls back to `$TMUX_PANE`), `SessionEntry.pipe_source_pane` is set to `Some("")` ŌĆö an empty-string sentinel that suppresses retries on every subsequent `Ask` turn. `cleanup_bg_windows()` skips `stop_pipe_pane` when the sentinel value is empty. Without this sentinel, a failed or inapplicable pipe-pane attempt would log a warning on every `Ask` turn for the lifetime of the session. All subsequent terminal output from that pane is appended to the log. `get_labeled_context()` reads the last 50 KB of the log (via `read_pipe_log()`) instead of `capture-pane` when the pane is not in a scrolled-back position ŌĆö giving the AI access to build output, test runs, and other verbose commands that have scrolled past the tmux scrollback buffer. The raw ANSI output is processed by `annotate_ansi()` (R2), which converts SGR foreground colour codes to semantic markers ŌĆö red (31, 91) ŌåÆ `[ERROR: text]`, yellow (33, 93) ŌåÆ `[WARN: text]`, green (32, 92) ŌåÆ `[OK: text]` ŌĆö and strips all remaining CSI, OSC, charset, and lone-ESC sequences, normalising `\r\n`. When no pipe log is available, the two `capture-pane` fallback paths also pass through `annotate_ansi()` via `capture_pane_with_escapes()` (at-bottom) and `capture_pane_at_scroll_with_escapes()` (scrolled position), both of which add `-e` to the `tmux capture-pane` call to preserve ANSI sequences before annotation. The log is stopped and deleted when the session is GC'd (`stop_pipe_pane()` in `cleanup_bg_windows()`) or when the daemon restarts (stale `de-pipe-*.log` files in `/tmp` are removed on startup). If pipe-pane is unavailable (old tmux, permission error), the daemon falls back silently to `capture_pane_with_escapes`.
- **Buffer-Based File Read** (`daemon/executor.rs`, N12): When the AI calls `read_file` with a `target_pane` that resolves to a **local** (non-SSH) pane, the daemon uses `tmux load-buffer` / `save-buffer` instead of pane scrollback to transfer file content. The command injected into the pane is `sed -n 'X,Yp' 'path' [| grep -E 'pat'] | tmux load-buffer -b 'de-rb-N' -; echo '__DE_DONE__'`. The daemon polls the pane with a 5-line capture until `__DE_DONE__` appears (200 ms interval, 30 s deadline), then reads the data with `tmux save-buffer -b 'de-rb-N' -` via subprocess and deletes the buffer. This bypasses the 600-line scrollback cap and is suitable for any file size. Buffer names use a monotonically-incrementing `BUFFER_COUNTER` (`daemon/session.rs`) to avoid collisions between concurrent tool calls. For **remote** target panes (SSH/mosh), the existing `build_remote_read_cmd` / `remote_run_and_capture` / `extract_marked` approach is retained (content limited to scrollback window size).
- **Session Cache** (`tmux/cache.rs`): A background task that polls every 2 seconds, captures all panes in the monitored session, and maintains a summarized snapshot. Used as a fallback when the client's specific pane is not available. `PaneState` carries `current_path` (shell CWD via `#{pane_current_path}`), `pane_title` (OSC terminal title via `#{pane_title}`), `scroll_position`, `history_size`, `in_copy_mode`, `synchronized`, `window_name` (the tmux window the pane belongs to), `dead` (true when the pane's foreground process has exited), `dead_status` (the exit code when `dead` is true), `last_activity: u64` (Unix timestamp from `#{pane_activity}` ŌĆö the last time the pane received a keypress or produced output), `start_cmd: String` (the command the pane was originally created with via `#{pane_start_command}`, N5 ŌĆö displayed in context and `list_panes` output when it differs from `current_cmd`), `pane_index: usize` (0-based window-relative pane index from `#{pane_index}` ŌĆö the number the user sees when pressing `ctrl+a q`; included in every context block header as `idx:N in 'window'` and shown in the tool-call approval prompt so users can visually map pane IDs to their tmux layout), and `shell_pid: u32` (PID of the foreground process from `#{pane_pid}`, refreshed every 2 s; the value present when the pane is idle is the shell's PID ŌĆö captured as `idle_pid` by `run_foreground()` before `send_keys` and used as the completion signal: command done when `pane_pid` returns to `idle_pid`). `SessionCache` carries an `environment` map (allowlisted keyŌåÆvalue pairs from `session_environment()`) refreshed each poll cycle; window topology data collected via `tmux list-windows` (7 fields: `window_id`, `window_name`, `pane_count`, `active`, `zoomed`, `last_active`, and `flags: String` from `#{window_flags}`); `bell_windows: RwLock<HashSet<String>>` tracking which window IDs are currently showing a bell; and `client_size: RwLock<(u16, u16)>` for viewport dimensions. `WindowState.has_bell()` returns true when the `#{window_flags}` string contains `!`; `has_activity()` returns true when it contains `#`. **Bell recovery**: on each 2 s `refresh()` cycle, newly-appeared `!` flags on non-daemon windows (windows not prefixed `de-bg-*`, `de-sj-*`, or `de-gs-*`) that are absent from `bell_windows` are logged to `events.jsonl` as `bell_recovery` events; the flag persists in tmux until the user visits the window, so bells are recoverable even after a daemon restart that missed the `alert-bell` hook. `set_active_pane(pane_id)` updates the active-pane state directly from an IPC hook notification, eliminating up to 2 s staleness when the user switches panes. `refresh_windows()` re-runs `tmux list-windows` immediately when the active window changes, triggered by the `session-window-changed` hook. `pane_map_summary(chat_pane)` generates a compact `[PANE MAP] idx:0=%3* (bash, 'main') | idx:1=%7 (vim, 'main')` line ŌĆö excluding daemon background windows and the chat pane; the active pane is marked with `*`. This line is injected into **every** AI turn prompt (first turn and subsequent turns) so the model always has a current indexŌåÆID mapping without calling any tool. `get_labeled_context()` prepends a `[SESSION ENVIRONMENT]` block when environment vars are present; a `[SESSION TOPOLOGY]` block showing window layout (including IDs, last-active flags, and `, bell!` / `, activity` annotations when `has_bell()`/`has_activity()` are set); appends `| cwd: /path` to `[ACTIVE PANE %N]` labels (no `idx:N` in the header ŌĆö window-relative indices live only in `[PANE MAP]`); classifies non-active panes into three labeled categories using `PaneState.window_name` (no extra tmux calls): `[VISIBLE PANE %N ŌĆö cmd ŌĆö cwd]` for panes in the same window as the chat interface, `[BACKGROUND PANE %N ŌĆö cmd ŌĆö cwd]` for daemon-launched windows (`de-bg-*`, `de-sj-*`, `de-gs-bg-*`, `de-gs-sj-*`, `de-gs-ir-*`); ghost shell windows additionally receive a `[ghost]` annotation; and `[SESSION PANE %N ŌĆö cmd ŌĆö cwd]` for other user-created windows (no `(idx:N in 'window')` in any of these headers). Includes path and title in all summary lines; appends `[dead: N]` when the pane's foreground process has exited with code N; annotates non-active panes with `[active Xs ago]`, `[idle Nm]`, or `[idle NhNm]` from `last_activity`; folds idle time into dead-pane tags; converts ANSI SGR colour codes via `annotate_ansi()`. The caller (`server.rs`) prepends a `[FOREGROUND TARGET] %N (cmd, 'window')` line to the context output so the AI sees the authoritative target pane ID before the topology blocks. When two or more tmux sessions exist, `get_labeled_context()` appends an `[OTHER SESSIONS]` block (N16) listing each non-current session's name, window count, activity age, attachment state, and any `bell!` / `activity` markers derived from a single `tmux list-windows -a` call in `list_session_flags()` (`tmux/session.rs`) ŌĆö one extra subprocess covers all other sessions at once. The `[OTHER SESSIONS]` block is appended only when the output already contains pane or environment data to avoid interfering with the `(no terminal context available)` fallback. Called on the first turn and by the `get_terminal_context` tool, which also appends a fresh `[PANE MAP]` line to its result. `refresh()` uses a single `list_panes_detailed()` call instead of N separate subprocesses. **Pane targeting guards** in `run_foreground()` (`executor/foreground.rs`): Two sequential checks run before the approval prompt. C3a (format guard) ŌĆö if the AI passes a `target_pane` that does not match `%\d+` (e.g. `"0"` or `"1"` instead of `"%7"`), the tool returns an error immediately with the session's `default_target_pane` so the model can self-correct on the same turn. C3b (stale-pane guard) ŌĆö if the pane ID is well-formed but no longer in the cache (pane closed, session changed), the tool returns an error containing the `default_target_pane` hint and the current `[PANE MAP]` so the model can re-anchor without a tool call. **Pane drift detection** in `server.rs`: at the start of each `Ask` turn, the incoming `client_target_pane` is compared to `SessionEntry.default_target_pane`; when they differ both are non-None, `default_target_pane` is updated to the new value and a `[Pane target changed]` `SystemMsg` is sent after `SessionInfo` (alongside the catch-up brief) so the model updates its mental model before generating its first token.
- **System Context Collector** (`sys_context.rs`): Runs once per daemon lifetime (via `OnceLock`). Captures OS release, uptime, memory, load average, top CPU processes, shell environment (curated safe variables only), and shell history. Prepended to the AI context on the first turn of each conversation.
- **Security & Data Masking Filter** (`ai/filter.rs`): Applied to all terminal context and user queries before transmission. Built-in patterns cover: AWS access key IDs, PEM private key blocks, GCP service-account JSON `"private_key"` fields, JWT bearer tokens, GitHub PATs (classic and fine-grained), database/broker connection URLs with embedded credentials, password/token/API-key assignments, URL query-param secrets, credit card numbers, and SSNs. Patterns are compiled once at daemon startup via `init_masking()`, which also incorporates any `extra_patterns` the user has added to `config.toml` ŌĆö built-in patterns cannot be disabled. Each compiled pattern entry is `(Regex, String, bool)` where the bool is `true` for user-configured patterns. Ten `AtomicUsize` counters track hits by type: `REDACT_AWS_KEY`, `REDACT_PRIVATE_KEY`, `REDACT_GCP_KEY`, `REDACT_JWT`, `REDACT_GITHUB_TOKEN`, `REDACT_DB_URL`, `REDACT_GENERIC`, `REDACT_CARD`, `REDACT_SSN`, and `REDACT_USER`; `mask_sensitive()` routes user-pattern hits to `REDACT_USER` and built-in pattern hits to the appropriate per-type counter via `counter_for()`. `get_redaction_counts()` returns a snapshot of all ten counters keyed by human-readable type name (all types always present, including zeros) and is included in the `DaemonStatus` response. 16 unit tests cover the full pattern set.
- **Prompt Engine Manager** (`config.rs`): Loads named system prompts from `~/.daemoneye/etc/prompts/<name>.toml`. Falls back to the compiled-in SRE prompt if no file is found. The built-in SRE prompt (compiled via `include_str!` from `assets/prompts/sre.toml`) instructs the AI that the terminal snapshot is not automatically included and to call `get_terminal_context` when needed, in addition to the `## Command Execution Modes` section. `config.toml` also carries a `[masking]` section (`MaskingConfig`) with an optional `extra_patterns` list. `AiConfig::context_window()` checks `context_window_tokens` (explicit override) first, then maps the configured model name to its context-window token limit (claude: 200k; gemini-1.5-pro: 2M; other gemini: 1M; gpt-4o/turbo: 128k; local/unknown: 32k) ŌĆö used by both the server-side budget warning and the CLI context-budget display. `AiConfig::effective_base_url()` resolves the API base URL with priority: explicit `base_url` config field ŌåÆ provider-specific default (ollama: `http://localhost:11434/v1`, lmstudio: `http://localhost:1234/v1`, openai: `$OPENAI_API_BASE` env var or `https://api.openai.com/v1`). `AiConfig::resolve_api_key()` returns a `"local"` dummy for `ollama` and `lmstudio` providers (no real key required).
- **LLM API Client** (`ai/mod.rs`, `ai/tools.rs`, `ai/backends/`): Implements `AiClient` trait for Anthropic, OpenAI/OpenAI-compatible (including Ollama and LM Studio), and Gemini with SSE streaming. `make_client(provider, api_key, model, base_url)` routes `"openai"`, `"ollama"`, and `"lmstudio"` to `OpenAiClient` and passes the resolved `base_url` directly ŌĆö `OPENAI_API_BASE` is no longer read inside the backend. Uses a process-wide `reqwest::Client` for connection reuse. Emits events over an unbounded channel for all supported tools: `Token`, `ToolCall` (command execution), `ScheduleCommand`, `ListSchedules`, `CancelSchedule`, `WriteScript`, `ListScripts`, `ReadScript`, `DeleteScript`, `WriteRunbook`, `DeleteRunbook`, `ReadRunbook`, `ListRunbooks`, `AddMemory`, `DeleteMemory`, `ReadMemory`, `ListMemories`, `SearchRepository`, `GetTerminalContext`, `ListPanes`, `Error`, and `Done`. Tool definitions and `dispatch_tool_event()` are in `ai/tools.rs`. The `chat()` method accepts a `use_tools: bool` parameter: when `false`, tool definitions are withheld and the model is constrained to plain-text output ŌĆö Gemini receives `toolConfig.functionCallingConfig.mode="NONE"` while Anthropic omits the `tools` field and OpenAI sets `tool_choice="none"`. This is used for webhook watchdog analysis calls to ensure the model emits `GHOST_TRIGGER: YES/NO` text rather than a function call. All three backends share the same `TOOLS` slice: Anthropic and OpenAI use `render_anthropic(TOOLS)` / `render_openai(TOOLS)`; Gemini uses `render_gemini(TOOLS)` (via the public `get_gemini_tool_definition()` function in `tools.rs`) ŌĆö the previously-inline `function_declarations` array in `gemini.rs` was replaced with this single shared source of truth. A `#[cfg(test)]` module in `tools.rs` validates that the Gemini renderer produces the correct count, order, name, type-case, and required-fields for every tool, catching any future divergence at `cargo test` time. **Extended thinking round-trips**: Both Gemini and Anthropic thinking models require the reasoning trace to be echoed back in subsequent turns. The Gemini backend extracts `thoughtSignature` from each `functionCall` part and re-emits it via `thoughtSignature` in `convert_messages`. The Anthropic backend collects `thinking_delta` and `signature_delta` SSE events, encodes the full `{"thinking": "...", "signature": "..."}` block as JSON in `ToolCall.thought_signature`, then in `convert_messages` emits it as a `type: "thinking"` content block immediately before the corresponding `tool_use` block ŌĆö satisfying Anthropic's multi-turn extended thinking requirement. `ToolResult` carries a `tool_name: String` field (populated at every construction site in `server.rs`) which the Gemini `convert_messages` uses as the `functionResponse.name` value ŌĆö ensuring the response is correctly attributed to the tool that was actually called rather than always mapping to `run_terminal_command`. Gemini thinking models occasionally emit Python-style function call syntax (e.g. `print(default_api.run_terminal_command(background = false, command = "...", target_pane = None))`) instead of a structured `functionCall` JSON block; the API signals this with `finishReason: MALFORMED_FUNCTION_CALL`. `parse_malformed_gemini_call()` recovers by extracting `command` and `background` from the argument list *inside* the `run_terminal_command(...)` parentheses only (using a paren-depth counter), preventing model commentary elsewhere in the message from accidentally matching; a `log::warn!` is emitted on every fallback invocation. The OpenAI backend SSE stream accumulates chunks in a `leftover` buffer until a newline is found; the buffer is capped at 1 MiB (`MAX_LEFTOVER_BYTES`) and an error is returned if exceeded to prevent memory exhaustion from a misbehaving proxy (S15). Note: local models served by Ollama or LM Studio vary widely in their tool-calling support ŌĆö models that do not support function calling will produce text responses only. **Circuit breaker** (F5): `send_with_retry` wraps `send_with_retry_inner` with a process-global `CircuitBreaker` (`consecutive_failures: AtomicU32`, `open_until: Mutex<Option<Instant>>`). After 5 consecutive failures the circuit opens for 60 s (fail-fast: returns an error immediately without sending). After the cooldown the circuit enters half-open state (one probe request allowed): a successful probe closes the circuit; a failed probe re-opens it. Current state (`"closed"` / `"open"` / `"half-open"`) is exposed via `circuit_state_str()` and included in `daemoneye status` output.
- **Schedule Store** (`scheduler.rs`): Thread-safe, file-backed store for scheduled jobs. Persistence is atomic: writes go to `.tmp` then rename over `~/.daemoneye/var/run/schedules.json`. Provides `add`, `cancel`, `list`, `take_due`, `mark_done` operations.
- **Scripts Module** (`scripts.rs`): Manages `~/.daemoneye/scripts/` ŌĆö executable scripts (chmod 700). Provides `list_scripts`, `write_script`, `read_script`, `delete_script`, `resolve_script` operations.
- **Runbook Module** (`runbook.rs`): Manages markdown runbook files in `~/.daemoneye/runbooks/<name>.md`. Parses YAML-style frontmatter to extract `tags: [...]` and `memories: [...]` fields. Validates content on write (requires `# Runbook:` heading and `## Alert Criteria` section). CRUD operations ŌĆö `load_runbook`, `write_runbook` (approval-gated), `delete_runbook` (approval-gated, warns if active jobs reference it), `list_runbooks` ŌĆö are exposed as AI tools. `watchdog_system_prompt()` loads any knowledge memory keys listed in `memories:` from `memory/knowledge/` and injects them as a `## Runbook Memory Context` block in the watchdog AI prompt.
- **Memory Module** (`memory.rs`): Provides persistent key-value storage under `~/.daemoneye/memory/` in three categories: `session/` (loaded into every AI turn as a `## Persistent Memory` block, capped at 32 KB), `knowledge/` (loaded on-demand by runbooks or `read_memory` tool), and `incidents/` (historical records, searchable only). CRUD operations ŌĆö `add_memory`, `update_memory`, `delete_memory`, `read_memory`, `list_memories` ŌĆö are exposed as AI tools without approval gates. `load_session_memory_block()` applies the sensitive-data masking filter and is called by `server.rs` on the first turn of every conversation. Memory entries support structured YAML frontmatter: `tags` (for retrieval matching), `summary` (one-line description shown in `list_memories` output and used for auto-search matching), `relates_to` (cross-references to other memory keys, runbooks, or scripts), `created`/`updated` (ISO timestamps managed automatically), and `expires` (optional TTL for time-bounded facts). `build_frontmatter()` serializes these fields consistently. `update_memory()` performs partial updates ŌĆö merging only the provided fields and preserving all others ŌĆö automatically setting the `updated` timestamp.
- **Manifest Module** (`manifest.rs`): Knowledge manifest builder and contextual auto-search. Three public functions: `build_knowledge_manifest()` ŌĆö compact 1 KB text index of all stored runbooks, scripts, knowledge memories, and incidents, shown as `## Available Knowledge` on the first AI turn; `auto_search_context(query, pane)` ŌĆö auto-loads up to 3 matching entries into `## Auto-loaded Knowledge` on the first turn, matching on names, tags, and summary words (Ōēź4 chars), then following `relates_to` links from matched knowledge memories to pull in related entries without a direct keyword hit; `related_knowledge_hints(output)` ŌĆö scans tool output and returns a `[Related knowledge: ŌĆ”]` hint line matching on runbook/memory names, tags, and summary words.
- **Search Module** (`search.rs`): Keyword search across all knowledge-base directories. `search_repository(query, kind, context_lines)` searches `runbooks/`, `scripts/`, `memory/{session,knowledge,incidents}/`, and the last 10,000 lines of `events.jsonl` depending on `kind` (`"runbooks"` \| `"scripts"` \| `"memory"` \| `"events"` \| `"all"`). Matches are case-insensitive; filenames are matched in addition to content. Results are capped at 50 and formatted with line numbers and surrounding context. Exposed as the `search_repository` AI tool.
- **Named Session Store** (`session_store.rs`, `header.rs`): Durable named conversation snapshots stored under `~/.daemoneye/var/sessions/<name>/` as `meta.toml` + `messages.jsonl`. An `index.json` file in `var/sessions/` tracks all saved sessions for fast listing. `SessionEntry` carries `saved_name: Option<String>`, `dirty: bool` (set on every message write-back, cleared on save/load), and `artifacts_created: Vec<ArtifactRef>` (accumulated as the AI creates runbooks, scripts, and memories during the session). On first save of a previously-unnamed session `backfill_session_origin()` walks `artifacts_created` and stamps a `session_origin: "<name>"` field into each artifact's frontmatter ŌĆö YAML frontmatter for memories and runbooks, comment-block header for scripts ŌĆö so artifacts are traceable back to the session that created them. Both injection helpers in `header.rs` are idempotent. Ghost sessions are excluded from `artifacts_created` tracking via an `is_ghost` guard. `daemoneye session import <id> --name <name>` is a standalone CLI subcommand (no daemon required) that reads an orphaned `var/log/sessions/<id>.jsonl` log and saves it as a named session.

### 2.4 Knowledge System

The knowledge system provides three inter-related persistence layers exposed to the AI as tools:

**Runbooks** (`runbook.rs`, `~/.daemoneye/runbooks/`): Markdown files with YAML-style frontmatter. Serve as watchdog procedures and environment-specific reference docs. Standard format:
```markdown
---
tags: [disk, storage]
memories: [disk_thresholds]
---
# Runbook: disk-check

## Purpose
ŌĆ”

## Alert Criteria
ŌĆ”

## Remediation Steps
ŌĆ”

## Notes
ŌĆ”
```
The AI uses `write_runbook` (approval-gated), `read_runbook`, `list_runbooks`, and `delete_runbook` (approval-gated). Before writing a new runbook the AI is instructed to call `list_runbooks` to avoid duplicates.

**Memory** (`memory.rs`, `~/.daemoneye/memory/`): Three-tier persistent key-value store. Each entry is a `.md` file in the appropriate category subdirectory. Entries support structured YAML frontmatter for richer retrieval and cross-referencing:

```markdown
---
tags: ["postgres", "database", "db", "pg"]
summary: "Primary/replica connection strings and lag thresholds"
relates_to: ["postgres-failover-runbook", "prod-db-hosts"]
created: "2026-01-15T10:00:00Z"
updated: "2026-03-31T09:00:00Z"
expires: "2026-12-31"
---
db1.internal:5432 (primary), db2.internal:5432 (replica)
```

- `session/` ŌĆö User preferences and recurring environment notes. Automatically loaded into every AI turn (via `load_session_memory_block()`), capped at 32 KB with masking applied.
- `knowledge/` ŌĆö Named service configs, host quirks, port tables. Loaded on-demand by watchdog runbooks (`memories:` frontmatter) or the `read_memory` tool. The `## Available Knowledge` manifest block (injected on the first AI turn) shows each entry as `key ŌĆö summary [tags]` / `key ŌĆö summary` / `key [tags]` / `key` depending on available metadata. `auto_search_context()` matches on names, tag words, and summary words (Ōēź4 chars), then follows `relates_to` links on matched entries to include related knowledge even without a direct keyword hit ŌĆö e.g. a query matching `db-hosts` can automatically surface `db-quirks` and `postgres-failover-runbook` via their `relates_to` fields.
- `incidents/` ŌĆö Historical incident records. Never auto-loaded; discovered via `search_repository`.

AI tools: `add_memory`, `update_memory` (partial field update ŌĆö preserves unspecified fields, auto-sets `updated` timestamp), `delete_memory`, `read_memory`, `list_memories` (output shows key and summary when available: `[knowledge] prod-db-hosts ŌĆö Primary/replica connection strings`).

**Manifest and auto-search** (`manifest.rs`): On the first turn, `server.rs` calls `build_knowledge_manifest()` (injected as `## Available Knowledge`) and `auto_search_context(first_query, pane_content)` (injected as `## Auto-loaded Knowledge` when matches are found). Auto-search scores matches by priority ŌĆö runbook name (0), runbook tag (1), knowledge key/summary/tag (2), script name (3) ŌĆö deduplicates, takes the top 3, then follows `relates_to` links on matched knowledge entries to fill remaining slots up to the 3-item cap. `related_knowledge_hints()` appends `[Related knowledge: runbook "ŌĆ”", memory "ŌĆ”"]` to tool results when the output text contains terms matching stored entries by name, tag, or summary word.

**Search** (`search.rs`): Cross-corpus keyword search. `search_repository(query, kind)` covers runbooks, scripts, all three memory tiers, and the event log. Results are grouped by file, annotated with line numbers and context, and capped at 50 matches.

**Session memory injection**: On the first turn of every conversation `server.rs` calls `load_session_memory_block()` and injects its output between the `## Execution Context` and `## Terminal Session` blocks. When no session memories exist the block is empty and the prompt format is unchanged.

### 2.5 Named Session Store

The session store (`src/session_store.rs`) provides durable, named snapshots of conversation history so sessions can be resumed across daemon restarts and shared by name.

**Storage layout** (`~/.daemoneye/var/sessions/`):
```
var/sessions/
  index.json               ŌåÉ lightweight index of all saved sessions
  postgres-incident/
    meta.toml              ŌåÉ name, description, turn_count, prompt, model, tags
    messages.jsonl         ŌåÉ full message history (one JSON object per line)
```

`save_session(name, ŌĆ”)` writes `meta.toml` and `messages.jsonl` atomically (tmp ŌåÆ `sync_all` ŌåÆ rename) then updates `index.json`. `load_session(name)` reads both files and returns `(SavedSessionMeta, Vec<Message>)`. `list_sessions()` reads only `index.json` for fast enumeration. `delete_session` and `rename_session` operate on the directory and update the index.

**`SessionEntry` integration** (`daemon/server.rs`):
- `saved_name: Option<String>` ŌĆö `None` for unnamed ephemeral sessions; set on `/session save` or auto-name.
- `dirty: bool` ŌĆö set to `true` on every message write-back; cleared to `false` after save or load.
- `artifacts_created: Vec<ArtifactRef>` ŌĆö records `{ kind, name, at_turn }` for every runbook, script, or memory the AI creates during the session.

**`session_origin` stamping** (`header.rs`, `session_store.rs`): When an artifact is created inside a named session, `executor/knowledge.rs` calls `inject_yaml_session_origin` (memories, runbooks) or `inject_comment_session_origin` (scripts) to add `session_origin: "<name>"` to the artifact's frontmatter or comment header. Both helpers are idempotent (no-op if the field already exists). On first-time save of a previously-unnamed session, `backfill_session_origin(artifacts, name)` walks `artifacts_created` retroactively and stamps each artifact that is missing the field. Script backfill preserves the 0700 permission by setting it on the temp file before rename.

**Auto-naming** (`daemon/server.rs`): When `[sessions] auto_name_enabled = true` (default) and the session crosses `auto_name_turn_threshold` (default 10) turns without a saved name, the daemon calls `suggest_session_name()` (a cheap `use_tools=false` model call) to generate a short kebab-case name, then sends a `SystemMsg` suggestion to the user in the chat pane.

**IPC protocol** (`src/ipc.rs`):
- `Request::{SaveSession, LoadSession, ListSavedSessions, DeleteSavedSession, RenameSavedSession, DiffSessions}`
- `Response::{SessionSaved, SessionLoaded, SavedSessionList, SessionDiff}`

**CLI slash commands**: `/session save [name]`, `/session tag [name]` (alias for save), `/session load <name>`, `/session list`, `/session delete <name>`, `/session rename <old> <new>`, `/session diff [name]`.

**Import subcommand**: `daemoneye session import <id> --name <name> [--desc "ŌĆ”"] [--force]` reads `~/.daemoneye/var/log/sessions/<id>.jsonl` without a running daemon and saves it as a named session using `session_store::save_session()`. Useful for retroactively naming sessions from previous incidents.

**Config** (`[sessions]` in `config.toml`):
```toml
[sessions]
auto_name_enabled = true
auto_name_turn_threshold = 10  # suggest a name after this many turns
load_recent_turns = 10         # turns loaded when resuming a saved session
```

### 2.6 Daemon Lifecycle

- **Startup**: `main()` is a plain synchronous function. For `daemoneye daemon` (without `--console`), `libc::fork()` is called *before* the tokio runtime starts ŌĆö the parent prints the child PID and exits; the child calls `libc::setsid()` and redirects stdin from `/dev/null`, then builds the tokio runtime. Inside the runtime: validates the API key (empty key is rejected with a fatal error; non-empty keys are also checked for the expected provider prefix ŌĆö Anthropic keys should start with `sk-ant-`, OpenAI keys with `sk-` ŌĆö and a `log::warn!` is emitted on mismatch so proxy setups and custom key formats still work), calls `init_masking()` with any user-defined `extra_patterns` to compile the masking pattern set (compile failures for both built-in and user-supplied patterns are logged via `log::warn!`), resolves the **initial session** ŌĆö a tmux session is always required; there is no degraded "no session" mode. Session resolution uses a two-step priority: (1) if the daemon is launched inside tmux (`$TMUX` is set), `detect_session()` is called (`tmux display-message -p '#S'`) and the current session is adopted (logs `INFO "Launched inside tmux ŌĆö adopting session 'ŌĆ”'."`); `managed_session` is set to `None` so the daemon does not own or recreate this session; (2) if launched outside tmux, the configured name is used ŌĆö CLI `--session NAME` override, then `[daemon] tmux_session` (default `"daemoneye"`) ŌĆö and `session_exists()` checks whether it already exists: if yes, adopt it (logs `INFO "Managed tmux session 'ŌĆ”' already exists ŌĆö adopting."`); if no, run `tmux new-session -d -s <name>` (logs `INFO "Created managed tmux session 'ŌĆ”'."`); a creation failure is a fatal startup error (`anyhow::bail!`). The resolved `managed_session: Arc<Option<String>>` (`Some` when the daemon owns the session, `None` when it adopted an existing one) is threaded into every `handle_client` invocation. When a `NotifySessionClosed` event arrives for the managed session name, the daemon recreates it automatically with `tmux new-session -d -s <name>` and repopulates `bg_session` and the cache so scheduled jobs and ghost shells resume without a restart. The new session fires the `after-new-session` global hook, which calls `install_session_hooks()` automatically. Starts the cache poller and session cleanup tasks, loads the schedule store from `~/.daemoneye/var/run/schedules.json` (if the primary path fails, an `ERROR` is logged and a transient in-memory store backed by a PID-unique temp file is used ŌĆö scheduled jobs work for the session but won't persist across restarts), checks for an already-running daemon (ping probe), then removes any stale socket file at `~/.daemoneye/var/run/daemoneye.sock` using `symlink_metadata()` (so a symlink at that path removes the symlink rather than its target) and binds the Unix socket. If `[webhook] enabled = true` in `config.toml`, the webhook HTTP server is spawned as an additional tokio task. All mutex lock sites use `.unwrap_or_log()` (the `UnpoisonExt` trait from `src/util.rs`) to recover from a poisoned lock rather than panicking; recovery is logged at ERROR level so lock-poison events are visible in diagnostics. If `ensure_dirs()` fails at startup, the daemon exits immediately with a non-zero status rather than continuing with broken I/O paths. **Systemd note**: `daemoneye daemon --console` (no fork) is required for `Type=simple` services; the generated service file uses this flag so systemd tracks the correct PID. When running as a systemd service the daemon starts outside tmux, so it creates the `"daemoneye"` session (or the configured name) automatically ŌĆö ghost shells, scheduled jobs, and webhook-triggered automation are available immediately with no interactive client connection required. `daemoneye chat` invoked outside of tmux will auto-attach to this session (opens a new `chat` window and exec-replaces itself with `tmux attach-session`).
- **Logging**: stdout and stderr are redirected to `~/.daemoneye/var/log/daemon.log` (or `--log-file FILE`) via `dup2` at startup. All diagnostic output goes through `env_logger` (keyed by `DAEMONEYE_LOG`; default `info`) so every line carries a UTC timestamp and level prefix. Startup errors that would prevent correct operation ŌĆö hook install failures, schedule store load failures, tmux command failures when `$TMUX` is set, masking pattern compile errors ŌĆö are logged at ERROR or WARN before the daemon enters the accept loop, ensuring they are always visible in `daemon.log`. Use `--console` to keep output on the terminal. View live with `daemoneye logs`.
- **Event log**: Written to `~/.daemoneye/var/log/events.jsonl` by default.
- **Task supervision** (A1): The four critical long-lived subsystem tasks (cache-poller, scheduler, session-cleanup, webhook) are started under `supervise()` rather than bare `tokio::spawn`. `supervise(name, shutdown, factory)` awaits each task's `JoinHandle`; on panic it logs ERROR and emits a `task_panic` event; on clean exit it logs WARN. Both paths wait an exponential backoff delay (1 s ŌåÆ 2 s ŌåÆ 4 s ŌåÆ 8 s ŌåÆ 16 s ŌåÆ 30 s cap; resets after 60 s of stable runtime) then call `factory()` again. Checks the `Arc<AtomicBool>` shutdown flag after each task exit and after each delay; exits without restarting when set. The flag is stored on the same `Arc` and set in both SIGTERM and SIGINT branches of the accept loop. Per-connection client handler tasks remain fire-and-forget (ephemeral by design).
- **Status**: `daemoneye status` sends a `Request::Status` IPC request. The daemon responds with `Response::DaemonStatus` containing: uptime in seconds (from `DAEMON_START: OnceLock<Instant>` recorded at startup), PID, number of active conversation sessions, provider/model, socket path, schedule count, and AI backend circuit breaker state. The CLI renders this as a human-readable two-column table (F1). The left column **Commands** section shows foreground and background command approval/denial counts alongside success/failure counts. The left column **Tooling** section shows, for each of Runbooks, Scripts, and File edits, the number of write approvals and denials since daemon start ŌĆö drawn from six atomic counters (`SCRIPTS_APPROVED/DENIED`, `RUNBOOKS_APPROVED/DENIED`, `FILE_EDITS_APPROVED/DENIED` in `daemon/stats.rs`) incremented at each approval gate in `executor/knowledge.rs` and `executor/file_ops.rs`. The right column includes a **Redactions** section listing per-type counts since daemon start, sorted by count descending; all built-in types are always shown even at zero, with `"User Defined:"` showing the aggregate count from any user-configured `[masking] extra_patterns`.
- **Shutdown**: `daemoneye stop` sends a `Shutdown` IPC request ŌĆö the daemon responds `Ok`, removes the socket file, and exits. SIGTERM and SIGINT are also handled gracefully via `tokio::signal::unix`. On signal, the shutdown sequence: (1) removes the socket so new clients fail fast; (2) uninstalls all four global tmux hooks (`pane-died`, `after-new-session`, `client-attached`, `client-detached`) via `set-hook -gu` so stale hooks do not fire against the dead daemon; (3) calls `cleanup_bg_windows()` on every active session (kills background windows, stops pipe-pane logs); (4) logs `"Daemon stopped cleanly."` (F2).

### 2.7 Scheduler and Watchdog

The scheduler runs as a background tokio task that polls `ScheduleStore::take_due()` every second. When a job fires, `run_scheduled_job()` is spawned:

1. **Action resolution**: `ActionOn::Ghost { runbook }` loads the named runbook, builds a `GhostConfig`, calls `GhostManager::start_session()`, and drives the ghost AI loop via `trigger_ghost_turn()` ŌĆö the same path used by webhook-triggered ghosts; `mark_done` is called after `trigger_ghost_turn` returns. `ActionOn::Script(name)` resolves to a full path via `scripts::resolve_script()`. `ActionOn::Command(cmd)` is used directly; an empty command string fast-fails immediately (logs an error and calls `mark_done` with failure) rather than timing out after 300 s with a misleading deprecation warning. `ActionOn::Alert` emits a `SystemMsg` and fires the notification hook without running any command.
2. **Execution**: A dedicated tmux window is created with `tmux::create_job_window()` using a temporary name, then renamed to `de-sj-<pane_num>-<unix_ts>-<cmd_slug>` (same convention as background windows) once the pane ID is known. The command is sent via `tmux send-keys` with `; exit $?` appended. `remain-on-exit on` is set so the pane stays alive after the process exits. The daemon subscribes to `BG_DONE_TX` via `bg_done_subscribe()` and waits for the `notify activity` IPC hook to fire (up to 300 s).
3. **Completion notifications**: `notify_job_completion()` is called after the job exits. It archives the pane scrollback to `pane_logs/` and sends a `SystemMsg` notification. Per FR-1.2.10: the window is killed immediately on success; on failure it is left open indefinitely for user inspection via `daemoneye schedule windows`.
4. **Watchdog AI analysis**: If the job has a `runbook` set, the captured output is passed to the configured LLM using `watchdog_system_prompt()` built from the runbook's context. If the AI response contains "ALERT", a `SystemMsg` is broadcast to connected clients and `fire_notification()` is called.
5. **Rescheduling**: `Every` jobs have their `next_run` advanced by `interval_secs` and transition back to `Pending`; `Once` jobs remain `Succeeded`/`Failed`.
6. **Notification hook**: `fire_notification()` runs the user-configured `[notifications] on_alert` shell command (from `config.toml`) with `$DAEMONEYE_JOB` and `$DAEMONEYE_MSG` environment variables.

### 2.8 Scripts Directory

`~/.daemoneye/scripts/` holds executable scripts managed by the daemon. Key properties:

- All scripts are written with `chmod 700` (owner-only, no group/other access).
- The AI can **write** scripts via the `write_script` tool, which triggers a `ScriptWritePrompt` IPC round-trip: the client displays the full script content and prompts the user to approve or deny. The daemon only writes the file on approval.
- The AI can **list** and **read** scripts without approval (read-only operations).
- The AI can **delete** scripts via the `delete_script` tool, which triggers a `ScriptDeletePrompt` IPC round-trip: the client confirms with the user before the file is removed.
- Scripts carry an optional inline comment header (`# --- daemoneye ---` ŌĆ” `# --- /daemoneye ---`) immediately after the shebang. The header holds `tags`, `summary`, `relates_to`, and any artifact-specific fields (e.g. `run_with_sudo: true`). `list_scripts` reads the first 4 KiB of each file to extract tags; `search_repository` matches header content via its normal grep path. The field names are identical to memory YAML frontmatter ŌĆö one format across scripts, runbooks, and memories.
- Scripts can be referenced by scheduled jobs (`ActionOn::Script(name)`) and are resolved to their full path at job execution time.
- Script names are validated to reject path traversal (no `/`, `\0`, `.`, or `..`).

### 2.9 Webhook Ingestion

The webhook server (`webhook.rs`) is an optional axum HTTP server spawned as a daemon-side tokio task when `[webhook] enabled = true` in `config.toml`. It is disabled by default.

**HTTP server**: Listens on `<bind_addr>:<port>` (default `127.0.0.1:9393`; set `[webhook] bind_addr = "0.0.0.0"` to expose on all interfaces). Two routes:
- `POST /webhook` ŌĆö alert ingestion
- `GET /health` ŌĆö liveness probe (returns `"ok"`)

**Authentication**: When `[webhook] secret` is non-empty, every `POST /webhook` request must include `Authorization: Bearer <secret>`. Requests without a valid token are rejected with `401 Unauthorized`.

**Payload format detection and parsers** (`parse_payload()`):
1. **Alertmanager / Grafana unified** ŌĆö detected by a top-level `"alerts"` array. Extracts per-alert `labels` (including `alertname`, `severity`), `annotations.summary`, `annotations.description`, `fingerprint` field (or SHA of sorted labels), and `status`.
2. **Grafana legacy** ŌĆö detected by a top-level `"state"` string without `"alerts"`. Maps `ruleName`/`title` ŌåÆ alert name, `message` ŌåÆ description, `"ok"` ŌåÆ Resolved.
3. **Generic fallback** ŌĆö tries `alertname`/`name`/`title` for the name, `severity`/`level`/`priority` for severity, `summary`/`message` for summary. If nothing matches, the full JSON body is used as the description.

**Processing pipeline** (runs asynchronously; handler returns `200 OK` immediately):
1. **Deduplication** ŌĆö alert fingerprint checked against `WebhookState.dedup` (a `Mutex<HashMap<fingerprint, last_seen_secs>>`). Suppressed if seen within `dedup_window_secs`. Fingerprint comes from the Alertmanager payload or is computed from sorted `key=value` label pairs.
2. **Masking** ŌĆö `mask_sensitive()` applied to `summary` and `description` before any further use.
3. **Formatting** ŌĆö human-readable single or multi-line message assembled from status, name, summary, description, and source.
4. **Event log** ŌĆö `log_event("webhook_alert", {...})` appends a record to `events.jsonl`.
5. **Session injection** ŌĆö `inject_into_sessions()` appends a `[Webhook Alert]` `Message` to every active session's JSONL file via `append_session_message()`.
6. **Chat pane notification** ŌĆö `notify_chat_panes()` calls `tmux display-message -d 8000 -t <chat_pane> "<first line>"` for every active chat pane.
7. **Severity gate** ŌĆö if `severity_rank(alert.severity) >= severity_rank(config.severity_threshold)`, `fire_notification()` is called (runs `[notifications] on_alert`). If `auto_analyze = true`, runbook analysis is also triggered.

**Runbook auto-analysis** (rate-limited per alert name within `dedup_window_secs`):
- `find_runbook_for_alert()` tries kebab-case, lower-case, and exact-case variants of the alert name (e.g. `"HighDiskUsage"` ŌåÆ `"high-disk-usage"`, `"highdiskusage"`, `"HighDiskUsage"`).
- If a runbook is found, `watchdog_system_prompt()` builds the watchdog prompt and the configured LLM analyses the formatted alert message.
- The AI response is always injected into all sessions, displayed in chat panes, and logged to `events.jsonl` as `"webhook_analysis"`.
- `parse_ghost_trigger()` scans the response for `GHOST_TRIGGER: YES` or `GHOST_TRIGGER: NO` (case-insensitive, last matching line wins) with fallback to a legacy `ALERT` keyword check. If `YES` and the runbook has `ghost_config.enabled = true`, `GhostManager::start_session()` is called in a spawned task to launch an autonomous remediation session. `start_session` writes the initial user message to the session JSONL file immediately so the file exists even if the ghost fails before its first AI turn, and logs a `ghost_start` record to `events.jsonl`.

**Ghost Shell policy** (`GhostPolicy::is_safe()`): non-sudo commands always pass; only scripts listed in `auto_approve_scripts` may use sudo ŌĆö regardless of `run_with_sudo`. `run_with_sudo: true` only affects `GhostPolicy::resolve_command()`, which auto-prepends `sudo` when executing an approved script, so the ghost AI can write `script.sh` instead of `sudo script.sh`; it does not grant permission to run arbitrary sudo commands. `trigger_ghost_turn` logs each turn start and tool dispatch to `daemon.log` (INFO), and writes `ghost_turn`, `ghost_complete`, and `ghost_error` structured records to `events.jsonl`. `inject_ghost_event` mirrors every lifecycle message to `events.jsonl` as `ghost_lifecycle` in addition to injecting it into session history. Background command output is archived to `~/.daemoneye/var/log/panes/` from the full pipe-pane log, bypassing the tmux scrollback cap.

## 4. Data Flow Example: Troubleshooting an Error

1. The user encounters a daemon failure in their active tmux pane.
2. The user runs `daemoneye ask "why did nginx crash?"` (or presses the tmux keybinding to open `daemoneye chat`).
3. The CLI client reads `$TMUX_PANE` from the environment and sends an `Ask` request over the Unix socket.
4. On the **first turn**, the daemon captures the last 200 lines from the client's pane via `get_labeled_context()`, runs the sensitive-data filter, and fetches the cached system context (or collects it fresh on the first-ever request). The context block begins with a `[FOREGROUND TARGET] %N (cmd, 'window')` line that pins the AI to the session's designated foreground pane ŌĆö the AI must copy this ID verbatim into `run_terminal_command` without deriving a pane from topology. Non-active panes are classified into `[VISIBLE PANE]` (same window as chat), `[BACKGROUND PANE]` (daemon-launched `de-bg-*` / `de-sj-*` / `de-gs-*`), and `[SESSION PANE]` (other user windows), each with working directory, OSC terminal title, synchronized-input flag, `[dead: N]` where applicable, and activity annotation (`[active Xs ago]` / `[idle Nm]` / `[idle NhNm]`). The active pane content has ANSI SGR colour codes converted to semantic markers (`[ERROR: ŌĆ”]` / `[WARN: ŌĆ”]` / `[OK: ŌĆ”]`) by `annotate_ansi()` (R2). High-signal tmux session environment variables are prepended as a `[SESSION ENVIRONMENT]` section. When two or more tmux sessions exist, an `[OTHER SESSIONS]` block is appended. A `[PANE MAP]` line (`idx:0=%3* bash | idx:1=%7 vim`) is injected on every turn (not just the first) so the AI always has a current indexŌåÆID mapping. On **subsequent turns** the full terminal snapshot is omitted from the user message ŌĆö only `[FOREGROUND TARGET]` and `[PANE MAP]` are injected alongside the token-budget note. The AI calls `get_terminal_context` when it needs a fresh pane snapshot.
5. The **Prompt Manager** supplies the SRE system prompt; the combined host context + terminal snapshot + user query is sent to the configured LLM API.
6. The API client streams tokens back; the daemon forwards each as a `Token` response to the client, which prints them as they arrive.
7. If the LLM invokes a tool call (e.g., `journalctl -u nginx.service`), the daemon sends a `ToolCallPrompt` response. For foreground commands the prompt includes `target_pane` (best-guess pane ID computed synchronously from cache). The client displays the command with its execution mode (`daemon ┬Ę runs silently` or `terminal ┬Ę visible to you`), and for foreground commands also shows the window-relative pane index (e.g. `ŌåÆ target: pane 1 in 'main' (%23)`) resolved via `tmux display-message`. Simultaneously the client applies a dark-blue highlight (`select-pane -P bg=colour17`) to the target pane so the user can immediately spot which pane will be used. The client waits up to 60 seconds for the user to approve (`y`) or deny; on denial the highlight is cleared immediately.
8. On approval:
   - *Background*: The daemon creates a dedicated tmux window (named `de-bg-<pane_num>-<unix_ts>-<cmd_slug>` after creation), detects the user's shell via `pane_current_command`, and sends the command wrapped with a `DAEMONEYE_EXIT_<id>:$?` marker (keeping the shell alive). `run_background_in_window` returns immediately with the pane ID; a background tokio task races two completion paths: (A) `pane-died` hook ŌåÆ capture output, archive to `pane_logs/`, inject `[Background Task Completed]` into session history, GC-kill the window; (B) exit marker found in scrollback ŌåÆ capture output, archive, inject `[Background Task Completed]`, leave window open for follow-up commands. A `tmux display-message` status banner is sent to the chat pane on either path. If the command requires `sudo`, the daemon first probes the credential cache (`sudo -n true`); if cached, no prompt is needed. Otherwise a `CredentialPrompt` IPC round-trip collects the password from the user (echo disabled in the chat pane); the credential is injected synchronously into the background pane after the `[de-sudo-prompt]` sentinel appears; auth failure is detected and logged at WARN.
   - *Foreground*: The daemon attaches a temporary `pane-title-changed` hook and injects the command via `tmux send-keys`. Completion detection uses a three-way branch: interactive commands (`ssh`, `mosh`, `telnet`, `screen`) return immediately once a shell-prompt pattern appears in the pane (up to 10 s), with a `[Interactive session started]` result and instructions to use `target_pane` for follow-up; remote panes use output-stability polling; local panes use event-driven hook + `pane_current_command` polling. If `sudo` is detected on a local pane, the daemon probes the credential cache and, if needed, prompts the user in the chat pane via `CredentialPrompt` (echo disabled, no focus switch) with up to 3 retry attempts; auth exhaustion or user timeout returns a structured error to the AI with an `install-sudoers` hint. After completion the hook is removed and 200 lines are captured.
   - The execution record is appended to `~/.daemoneye/var/log/events.jsonl`.
9. The LLM receives the tool result and continues its response. This loop repeats until the LLM produces a final answer with no further tool calls.
10. The daemon sends `Ok` to signal completion. The conversation history is stored under the session ID for the next turn.
