# Product Requirements Document (PRD) Details for DaemonEye

This document specifies the functional and non-functional requirements for DaemonEye, building upon the Product Definition.

## 1. Functional Requirements

### 1.1 Daemon Process & tmux Integration

- **FR-1.1.1**: The application MUST run as a background daemon process, independent of any specific terminal emulator.
- **FR-1.1.2**: The application MUST use `tmux` as its presentation layer and session interaction mechanism.
- **FR-1.1.3**: The daemon MUST be capable of spawning new tmux panes within the user's active tmux session to present the AI interface.
- **FR-1.1.4**: The application MUST provide a CLI tool or tmux keybinding to trigger the AI agent, which communicates with the background daemon.
- **FR-1.1.5**: The daemon MUST append a structured JSON event record to `~/.daemoneye/events.jsonl` for every tool call execution event (approved, denied, or timed out).

### 1.2 AI Agent Integration

- **FR-1.2.1**: The AI agent MUST act as an expert sysadmin and security expert, capable of determining the right tools to use.
- **FR-1.2.2**: When activated, the application MUST capture terminal context (visible output, backscroll history, environment state) from the currently active tmux pane and provide it to the AI agent on the **first turn** of each session. On subsequent turns the terminal snapshot is NOT automatically injected; the AI MUST request it explicitly via the `get_terminal_context` tool when it needs a fresh view of the screen. This eliminates repeated terminal snapshot accumulation in conversation history and reduces token consumption in conversational sessions. The captured context MUST include, for each pane: (a) the shell's current working directory (`#{pane_current_path}`); (b) the OSC terminal title (`#{pane_title}`) when set by a running application (e.g., vim, ssh, k9s) and distinct from the command name; (c) a `[dead: N]` annotation for background panes whose foreground process has exited with code N (remain-on-exit mode); when the pane has been idle since completion, the idle duration MUST be folded into the same tag as `[dead: N, idle Xs]` / `[dead: N, idle Nm]` / `[dead: N, idle NhNm]` so the AI can see both exit code and inactivity age in a single annotation. Non-active panes MUST be classified into three labeled categories: `[VISIBLE PANE]` (same tmux window as the chat interface), `[BACKGROUND PANE]` (daemon-launched `de-bg-*` / `de-sched-*` windows), and `[SESSION PANE]` (other user-created windows). This classification MUST use the cached `window_name` field with no additional tmux subprocess calls. (d) a temporal activity annotation on non-active panes: `[active Xs ago]` when last activity was under 30 s ago, `[idle Nm]` when under 1 h, and `[idle NhNm]` otherwise — derived from `#{pane_activity}` (Unix timestamp of last keypress or output in the pane). (e) ANSI SGR foreground colour codes in the active pane content MUST be converted to semantic markers before the context is sent to the AI: red (codes 31, 91) → `[ERROR: text]`, yellow (33, 93) → `[WARN: text]`, green (32, 92) → `[OK: text]`; all other escape sequences MUST be stripped. This applies to both the pipe-log path (via `annotate_ansi()`) and the `capture-pane` fallback paths (via `capture_pane_with_escapes()` with `-e` flag, then `annotate_ansi()`). (f) when two or more tmux sessions exist, an `[OTHER SESSIONS]` block MUST be appended to the context snapshot listing each non-current session's name, window count, time since last activity, and attachment state — derived from `tmux list-sessions -F` (N16). This block MUST appear only when the snapshot already contains pane or environment data (i.e. MUST NOT convert a `(no terminal context available)` result into a non-empty snapshot). Additionally, the application MUST fetch high-signal tmux session environment variables via `tmux show-environment` filtered against an allowlist (AWS_PROFILE, AWS_DEFAULT_REGION, AWS_REGION, KUBECONFIG, KUBE_CONTEXT, KUBECTL_CONTEXT, VAULT_ADDR, DOCKER_HOST, DOCKER_CONTEXT, ENVIRONMENT, APP_ENV, NODE_ENV, RAILS_ENV, RACK_ENV, VIRTUAL_ENV, CONDA_DEFAULT_ENV, GOPATH, GOENV, JAVA_HOME, LANG, LC_ALL) and prepend them as a `[SESSION ENVIRONMENT]` block to every AI context snapshot.
- **FR-1.2.3**: The AI MUST be able to analyze stack traces, crash logs, failing services, and security scan outputs to provide root cause analysis and remediation strategies.
- **FR-1.2.4**: The AI interacts directly with the user's active terminal using tmux session features. This allows the AI agent to "pair program" with the user. By hooking into the user's terminal session via tmux, the AI agent can execute commands, read output, and respond to system prompts with the user's permission.
- **FR-1.2.5**: The application MUST actively audit the system state (OS release, uptime, memory, load average, top CPU processes, shell environment, and shell history) once per session, cache it, and prepend this summary to the AI agent's context alongside the visible terminal buffer.
- **FR-1.2.6**: The daemon MUST detect its own hostname and whether the user's active tmux pane is connected to a remote host (via SSH or mosh). This execution context MUST be injected into the first-turn AI prompt so the AI understands which machine each execution mode (`background` vs `foreground`) will target.
- **FR-1.2.7**: The application MUST support two distinct command execution modes and the AI MUST be instructed to choose between them appropriately:
  - *Background mode*: Command runs in a dedicated tmux window (`de-bg-<session_name>-<YYYYMMDDHHMMSS>-<id_short>`) on the daemon host using the user's configured shell. The window is configured with `remain-on-exit on`. The command is wrapped with a shell-appropriate IPC notification (`cmd; __de_ec=$?; daemoneye notify complete <pane_id> $__de_ec <session>` for POSIX shells; `set __de_ec $status` for fish/csh) so the shell remains alive after the command finishes, keeping the window available for follow-up commands. The tool call returns **immediately** with the assigned pane ID; completion is detected asynchronously via two paths: (primary) the command wrapper sends the exit code via `daemoneye notify complete` IPC, broadcasting on `COMPLETE_TX` — window remains open for reuse; (fallback) the shell exits unexpectedly, `pane-died` hook fires and broadcasts on `BG_DONE_TX` — window is GC'd automatically. Upon completion the pane's entire scrollback history is logged to `~/.daemoneye/pane_logs/` and a `[Background Task Completed]` context message (containing exit code and captured output) is injected into the AI session history. Up to 5 background windows persist per chat session; the oldest completed window is evicted when the cap is reached; all windows are closed when the session expires. The AI MUST be able to explicitly close a background window via a `close_background_window(pane_id)` tool call (no user approval required). The daemon MUST automatically garbage-collect background windows whose `completed_at` timestamp is more than 15 minutes old, checked on the 60 s session-cleanup cycle, to prevent window accumulation even if the AI never calls `close_background_window`. The application MUST share background commands with the user and gain approval before executing them.
  - *Foreground mode*: Command is injected into the user's active tmux pane via `send-keys`. Completion MUST be detected via a three-way branch keyed on the command type: (1) *Interactive commands* (`ssh`, `mosh`, `telnet`, `screen`, and similar session-hijacking tools) — the daemon MUST detect these via `is_interactive_command()` before injecting the command. It MUST wait up to 10 seconds for a shell-prompt pattern to appear in the pane output, then return immediately with an `[Interactive session started]` tool result identifying the destination host and instructing the AI to use `run_terminal_command(target_pane=...)` for subsequent commands. It MUST NOT wait for the interactive session to exit. Non-interactive sub-cases (`ssh host command`, `ssh -N`) are excluded and handled by path (2) or (3) as appropriate. (2) *Remote panes* (pane whose foreground process is `ssh` or `mosh-client`) — output-stability polling (two consecutive identical snapshots, 500 ms interval, 30 s timeout). (3) *Local panes* — event-driven `pane-title-changed` hook notification with `pane_current_command` polling fallback (45 s timeout). The user can see and interact with the command as it runs. If the user's pane is SSH'd to a remote host, the command executes on that remote host. If the AI does not explicitly set a target pane, the CLI client MUST perform automatic pane discovery before connecting: auto-select the sole sibling pane in the same window, present a numbered picker when multiple siblings exist, or offer to split the window side-by-side (default) or pick from other session panes when the chat pane is alone. The chosen pane MUST be persisted to `~/.daemoneye/pane_prefs.json` keyed by session name so the user is never prompted again after the first run. The persisted preference is always overridable by explicit AI instructions. *Exit code capture*: after detecting command completion, the daemon MUST read the tmux session environment key `DE_EXIT_<N>` (where N is the numeric pane ID, e.g. `DE_EXIT_3` for pane `%3`) via `tmux show-environment` to obtain the real exit code. This key is written by an optional shell hook (`PROMPT_COMMAND` / `precmd_functions`) that runs after every command completes. If the key is absent (hook not configured), the daemon MUST record exit code 0. The daemon MUST call `stats::finish_command(cmd_id, exit_code)` to update the recent-command record and increment the appropriate `COMMANDS_FG_SUCCEEDED` / `COMMANDS_FG_FAILED` counter. `daemoneye setup` MUST print the bash and zsh hook snippets so users can opt in.
- **FR-1.2.10**: The daemon MUST support scheduling commands to run once at a specified UTC datetime or repeatedly at a specified ISO 8601 interval (e.g. `PT5M`, `PT1H`, `P1D`). Scheduled jobs MUST persist across daemon restarts in `~/.daemoneye/schedules.json` using atomic writes. Each job MUST run in a dedicated tmux window pane (`de-sched-<YYYYMMDDHHMMSS>-<id_short>`), which MUST be destroyed on success and left in place on failure for inspection. Output is captured vis 'capture-pane' and the exit code is extracted natively via `pane_dead_status` and returned to the AI. Scheduled job panes are tracked via `pane-died` and `alert-bell` hooks. The AI MUST be able to create, list, cancel, and delete scheduled jobs via tool calls. The daemon MUST support a `ScheduleWritePrompt` approval gate that displays the full schedule content to the user before writing.
- **FR-1.2.11**: The daemon MUST support watchdog jobs that run a command on a schedule and pass the captured output to the AI for analysis using a named runbook (markdown file in `~/.daemoneye/runbooks/<name>.md`). Runbooks use YAML-style frontmatter to declare tags and knowledge memory references. The watchdog AI prompt MUST include any knowledge memory entries referenced in the runbook's `memories:` frontmatter field. When the AI detects an alert condition, it MUST send a `SystemMsg` notification to connected clients. If `[notifications] on_alert` is configured in `config.toml`, the daemon MUST invoke that shell command with `$DAEMONEYE_JOB` and `$DAEMONEYE_MSG` environment variables set.
- **FR-1.2.12**: The daemon MUST maintain a `~/.daemoneye/scripts/` directory of executable scripts (mode 0700). The AI MUST be able to write new scripts and update existing ones, subject to a `ScriptWritePrompt` approval gate that displays the full script content to the user before writing. The AI MUST be able to list available scripts and read their contents without approval.
- **FR-1.2.13**: The daemon MUST provide a Knowledge System comprising three inter-related persistence layers accessible as AI tools:
  - *Runbook CRUD*: The AI MUST be able to `write_runbook` (approval-gated, validates required sections), `read_runbook`, `list_runbooks`, and `delete_runbook` (approval-gated, warns if active scheduled jobs reference it). Runbooks MUST use markdown format with YAML-style frontmatter (`tags`, `memories` fields). `delete_runbook` MUST check active jobs and include affected job names in the approval prompt.
  - *Persistent Memory*: The AI MUST be able to `add_memory`, `delete_memory`, `read_memory`, and `list_memories` across three categories — `session` (auto-loaded at each turn, 32 KB cap, masking applied), `knowledge` (on-demand via runbook references or explicit reads), and `incident` (historical records, search-only). Memory operations MUST NOT require user approval.
  - *Repository Search*: The AI MUST be able to `search_repository` across runbooks, scripts, all memory categories, and the last 10,000 lines of `events.jsonl`. Search MUST be case-insensitive, match both filenames and content, include configurable context lines around each match, and cap results at 50 entries.
  - *Terminal Context*: The AI MUST be able to call `get_terminal_context` to obtain a fresh snapshot of the current tmux session (active pane contents, background pane summaries, session topology, and environment). This tool requires no user approval. It MUST be the mechanism by which the AI accesses the terminal on turns after the first.
  - *Pane Discovery*: The AI MUST be able to call `list_panes` to obtain a compact table of all active panes in the session (pane ID, window name, foreground command, working directory, terminal title, synchronized and dead-process flags). The chat pane MUST be excluded from results. This tool MUST read from the in-memory SessionCache and MUST NOT spawn a tmux subprocess. Each entry MUST include the activity annotation (same `[active Xs ago]` / `[idle Nm]` / `[idle NhNm]` format used in the terminal context snapshot) so the AI can identify recently-used panes. The agent uses this to identify a pane by its characteristics before targeting it via `run_terminal_command(target_pane=...)`.
- **FR-1.2.9**: The chat input MUST operate in raw terminal mode to support readline-style editing. Users MUST be able to navigate the current input line with ←/→ arrow keys (and Home/End, Ctrl+A/E) and submit past entries with ↑/↓ arrow keys. The input MUST support in-line deletion (Backspace, Delete, Ctrl+K, Ctrl+U). The input area MUST word-wrap long text across multiple rows (up to a maximum of 5) rather than scrolling horizontally; the input box MUST grow upward dynamically as the buffer fills and collapse back to 1 row when text is deleted or on submission. The top border of the input frame MUST display the local `user@host` identity. History MUST persist for the lifetime of the chat session and reset when the session ends.

- **FR-1.2.14**: The daemon MUST optionally expose an HTTP webhook endpoint (configurable port, default 9393; bind address configurable via `[webhook] bind_addr`, defaulting to `"127.0.0.1"` so that the endpoint is not exposed to the network on internet-facing hosts; set to `"0.0.0.0"` to expose on all interfaces; disabled by default). It MUST accept Prometheus Alertmanager v4 format, Grafana unified alerting format, Grafana legacy webhook format, and a generic JSON fallback. Received alerts MUST be: (a) deduplicated within a configurable window by fingerprint (sourced from the payload or computed from sorted label key=value pairs); (b) masked for sensitive data via the existing `mask_sensitive()` filter before any further processing; (c) injected as `[Webhook Alert]` messages into all active AI session JSONL files; (d) displayed via `tmux display-message` in all active chat panes. When `[webhook] severity_threshold` is met, `fire_notification()` MUST be invoked. When `[webhook] auto_analyze = true` and a matching runbook exists (checked using kebab-case, lowercase, and exact-case variants of the alert name), a runbook-based AI analysis MUST run; if the AI response contains "ALERT" the analysis MUST be injected into sessions and chat panes. A configurable Bearer token MUST be validated when `[webhook] secret` is non-empty; requests without a valid token MUST be rejected with `401 Unauthorized`. The endpoint MUST return `200 OK` immediately (processing is asynchronous). A `GET /health` endpoint MUST return `"ok"` for liveness probes.

- **FR-1.2.15**: The daemon MUST detect client-attach and client-detach events via two global tmux hooks (`client-attached` and `client-detached`) installed at startup. On `client-detached`, the daemon MUST record `last_detach = now` and `messages_at_detach = messages.len()` on every active `SessionEntry` whose `tmux_session` field matches the detached session. On `client-attached`, the daemon MUST clear `last_detach` on matching entries. When the next `Ask` request arrives and `away_secs >= 30`, the daemon MUST scan `messages[messages_at_detach..]` for event messages (identified by the prefixes `[Background Task Completed`, `[Webhook Alert]`, `[Watchdog]`, `[Watch Pane`) and, if any are found, send a `Response::SystemMsg` containing a `[Catch-up] N events while you were away (Xm): …` summary immediately after `Response::SessionInfo`. The catch-up brief MUST be sent only once per detach cycle (i.e. `last_detach` MUST be cleared after generating the brief).

- **FR-1.2.16**: When the user's tmux environment contains two or more sessions, the daemon MUST include an `[OTHER SESSIONS]` block in every terminal context snapshot (both automatic first-turn and on-demand `get_terminal_context` calls). Each entry MUST include the session name, window count, time elapsed since the session's last activity (formatted as `Xm ago`, `XhYm ago`, etc.), and whether a client is currently attached. The block MUST be omitted when the snapshot has no pane or environment data (to prevent polluting the `(no terminal context available)` fallback path during tests or when the tmux session is unavailable).

### 1.3 Prompt Library

- **FR-1.3.1**: The application MUST include a library of pre-defined prompts for common tasks.
- **FR-1.3.2**: Users MUST be able to create, save, and manage their own custom prompts. A `daemoneye prompts` subcommand MUST list available prompts with a brief preview of each prompt's system message. A `/prompt <name>` in-chat command MUST start a new session using the named prompt as the system message. *(Config and file loading are implemented; the subcommand and in-chat command are pending.)*
- **FR-1.3.3**: All user-defined and standard prompts MUST be stored in the user's home directory under `~/.daemoneye/prompts`.

### 1.4 Authentication & Security

- **FR-1.4.1**: Sensitive data (passwords, secret keys, PII) MUST be masked or filtered from the terminal buffer before being transmitted to the AI API. The filter MUST cover at minimum: AWS access key IDs, PEM private key blocks, GCP service-account JSON private key fields, JWT bearer tokens, GitHub personal access tokens, database/broker connection URLs with embedded credentials, password/token/API-key assignments, URL query-param secrets, credit card numbers, and SSNs. Users MUST be able to extend the filter with additional patterns via `[masking] extra_patterns` in `config.toml`; built-in patterns MUST NOT be disableable. Redaction counts MUST be tracked per type using per-process atomic counters and exposed in `daemoneye status` under a **Redactions** section. All built-in counter types MUST always be displayed, including those with a zero count. Hits from user-configured `extra_patterns` MUST be aggregated into a separate `"User Defined"` counter distinct from all built-in type counters.
- **FR-1.4.2**: Users MUST have explicit controls over what terminal context is sent to the LLM.
- **FR-1.4.3**: When the AI proposes a background command that requires `sudo`, the daemon MUST detect the sudo password prompt in the tmux window via `capture-pane`. It MUST then send a `CredentialPrompt` IPC response to the client (echo disabled). The password MUST be injected into the background window via `send-keys` and MUST NOT be logged, stored on disk, or transmitted to the AI.
- **FR-1.4.4**: When the AI proposes a foreground command that requires `sudo`, the daemon MUST inject the command into the user's terminal pane and wait briefly for it to start. The daemon MUST then capture the pane output and check whether a sudo password prompt is present. If a password prompt is detected: the application MUST notify the user in the chat interface, make the target terminal pane active so the user can type their password, wait for the password to be entered and the command to complete, then switch focus back to the AI chat pane. If no password prompt is detected (e.g. a NOPASSWD sudoers rule or a still-valid credential timestamp), the application MUST proceed with a standard wait interval without switching panes.

### 1.5 Extensibility

- **FR-1.5.1**: The application MUST include a native plugin architecture for community extensions.
- **FR-1.5.2**: The application MUST allow plugins to hook into AI prompt lifecycles, and third-party APIs (e.g., AWS/GCP/Azure).

## 2. Non-Functional Requirements

### 2.1 Performance

- **NFR-2.1.1**: Capturing tmux buffers and transmitting to the AI MUST NOT block the user's terminal interaction.
- **NFR-2.1.2**: The daemon MUST be lightweight and consume minimal background system resources when idle.

### 2.2 Compatibility & Environment

- **NFR-2.2.1**: The application MUST run on standard modern Linux distributions (Ubuntu, Fedora, Arch, etc.).
- **NFR-2.2.2**: The application requires `tmux` to be available in the system PATH.

### 2.3 Usability

- **NFR-2.3.1**: The interaction with the AI agent inside the tmux pane MUST feel natural and responsive.
- **NFR-2.3.2**: The chat interface MUST provide a visual indicator (animated spinner) while waiting for the AI to respond.
- **NFR-2.3.5**: The chat interface MUST display a context-budget indicator in the bottom border of each user query box, showing `turn N · Xk / Yk tokens · Z% remaining`. The context window (Yk) MUST be determined from the configured model. The indicator MUST be color-coded by percentage consumed: dim when under 50 %, yellow at 50–74 %, bold red at ≥ 75 %. The value MUST persist across turns (carried from the previous AI response's `UsageUpdate`) so the user always sees an accurate budget before they submit the next query.
- **NFR-2.3.3**: Tool call prompts MUST clearly communicate which execution mode will be used and where the command will run (daemon host or user's terminal pane).
- **NFR-2.3.4**: The chat interface SHOULD use color and Unicode symbols to visually distinguish AI responses, tool calls, approvals, errors, and system messages.
